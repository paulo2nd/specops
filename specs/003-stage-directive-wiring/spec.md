# Feature Specification: Stage-Wide Directive Wiring

**Feature Branch**: `003-stage-directive-wiring`

**Created**: 2026-07-05

**Status**: Draft

**Input**: User description: "Wire the SpecOps directives across every Speckit stage so the phase state machine (SPECIFY → PLAN → TASKS → IMPLEMENT → REVIEW → DONE) is driven end-to-end by the injected agent prompts, not by the human remembering to run CLI commands."

## Overview

Today `specops init` injects directive blocks into only two Speckit prompts —
**plan** and **implement** — and installs the `/specops-review` command. The
SpecOps ledger, however, models a full phase state machine
(`SPECIFY → PLAN → TASKS → IMPLEMENT → REVIEW → DONE`). Several seams of that
machine are unwired for the agent, so an agent following only the injected
prompts cannot drive a feature end to end. The gaps are:

- The ledger is never created by any prompt — nothing tells the agent to run
  `specops status init-spec`, so the first `start-task` in the implement loop
  fails if the human forgot to create it manually.
- Phase transitions between stages are never triggered by any prompt (only
  `/specops-review` transitions `REVIEW → DONE|IMPLEMENT`), so the ledger phase
  goes stale relative to the work actually happening.
- The review cycle is never opened (nothing runs `transition-phase REVIEW`),
  which the review command depends on to record its outcome.
- The rule that every `tasks.md` line must carry `[SC-xxx]` coverage tags lives
  in the **plan** directive, but `tasks.md` is generated by the **tasks** prompt
  which receives no injection — so the rule is invisible where it applies, yet
  `specops consistency` later enforces it.
- The **specify** and **tasks** stages receive no SpecOps guidance at all.

This feature closes those seams by extending SpecOps' existing injection
mechanism so the agent-facing prompts carry the ledger and phase commands at the
correct moments, while preserving SpecOps' non-negotiable constraint of being an
additive, reversible layer over Speckit.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Ledger exists when implementation starts (Priority: P1)

A developer runs `specops init` on their Speckit repository, then drives a
feature with an agent through specify → plan → tasks → implement. When the agent
reaches the implement stage, the execution ledger already exists and is
populated with the feature's task IDs — the agent's first `start-task` succeeds
without the developer having manually run any ledger command.

**Why this priority**: This is the highest-impact discontinuity. Without it the
documented implement loop fails on its very first command in the common case
where the human did not know to create the ledger. Fixing it alone makes the
happy path work.

**Independent Test**: In a SpecOps-initialized Speckit repo, generate a
`tasks.md` via the tasks stage, then run the implement stage as an agent would.
Verify the ledger was created (populated with the task IDs from `tasks.md`) and
that the first task transition to `IN_PROGRESS` succeeds, all without a human
issuing a ledger command.

**Acceptance Scenarios**:

1. **Given** a SpecOps-initialized repo with a completed `tasks.md` and no
   ledger yet, **When** the agent completes the tasks stage, **Then** the ledger
   is created and contains every task ID from `tasks.md` as `PENDING`.
2. **Given** a ledger already created for the feature, **When** the tasks stage
   runs again (regeneration), **Then** the agent does not error on the existing
   ledger and the developer is not blocked.
3. **Given** the created ledger, **When** the agent begins the implement stage,
   **Then** the first `start-task` succeeds with no manual human command.

---

### User Story 2 - Ledger phase reflects the current stage automatically (Priority: P2)

As an agent advances a feature through the Speckit lifecycle, the ledger's phase
field tracks the stage the work is actually in, and the review cycle is opened
automatically when implementation finishes — so a `status show` at any point
reports the true phase and `/specops-review` finds an open cycle to record its
outcome into, without a human running `transition-phase` by hand.

**Why this priority**: Once the ledger exists (US1), keeping its phase truthful
is what makes `status show`, review gating, and any phase-based automation
meaningful. It is second because the machine is usable (tasks can be tracked and
completed) even if the phase label lags.

**Independent Test**: Drive a feature through the stages as an agent and, after
each stage, inspect the ledger phase; confirm it matches the active stage and
that an open review cycle exists before `/specops-review` runs.

**Acceptance Scenarios**:

1. **Given** the agent has finished the plan stage, **When** it moves to the
   tasks stage, **Then** the ledger phase reflects the progression toward
   implementation rather than remaining at an earlier phase.
2. **Given** the agent has completed all implementation tasks, **When** it hands
   off for review, **Then** a review cycle is open and the ledger phase is
   `REVIEW`.
3. **Given** an open review cycle, **When** `/specops-review` completes, **Then**
   it records `APPROVED`/`REJECTED` into that cycle exactly as it does today (no
   regression to existing review behavior).

---

### User Story 3 - Coverage tags are required where tasks are generated (Priority: P3)

When an agent generates `tasks.md` in the tasks stage, it is instructed — in the
tasks stage itself — to tag every task line with the `[SC-xxx]` Success Criteria
it covers, using only IDs that exist in the spec. As a result `specops
consistency` passes on freshly generated tasks without a human editing
`tasks.md` after the fact.

**Why this priority**: This removes a latent inconsistency (a rule enforced by
`consistency` but never shown at the point of generation). It is lowest priority
because it degrades gracefully — a human can still add tags manually — and it
does not block the core ledger flow.

**Independent Test**: Generate `tasks.md` as an agent in a SpecOps-initialized
repo, then run `specops consistency`; confirm it passes with no missing-tag
violations and that every tag references an existing spec Success Criterion.

**Acceptance Scenarios**:

1. **Given** a spec with Success Criteria `SC-001..SC-00N`, **When** the agent
   generates `tasks.md`, **Then** every task line carries at least one valid
   `[SC-xxx]` tag drawn only from those IDs.
2. **Given** freshly generated `tasks.md`, **When** `specops consistency` runs,
   **Then** it reports no coverage violations.

### Edge Cases

- **SpecOps not initialized**: In a plain Speckit repo (no `specops.json`, no
  SpecOps CLI available), every injected stage prompt MUST still complete its
  core Speckit work. SpecOps steps degrade to a no-op with, at most, a single
  informational line — never a hard failure.
- **Ledger already exists**: Re-running the tasks stage (or re-attempting ledger
  creation) MUST NOT abort the agent's stage on an "already exists" condition.
- **Out-of-order stages**: If the agent reaches a stage whose expected
  predecessor phase is not recorded (e.g., the human skipped a stage), the phase
  step MUST fail safe and surface the condition rather than corrupting the
  ledger.
- **Speckit upgrade removes blocks**: A Speckit upgrade that rewrites prompt
  files may strip the injected blocks; re-running `specops init` MUST re-inject
  all stage blocks idempotently (same guarantee that exists today for plan and
  implement).
- **Uninstall**: Removing SpecOps MUST restore every touched prompt to a
  byte-identical pre-injection state.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `specops init` MUST inject a SpecOps directive block into the
  Speckit **tasks** prompt, in addition to the existing plan and implement
  prompts.
- **FR-002**: `specops init` MUST inject a SpecOps directive block into the
  Speckit **specify** prompt.
- **FR-003**: The tasks-stage directive MUST instruct the agent to tag every
  generated `tasks.md` task line with one or more `[SC-xxx]` coverage tags,
  using only Success Criteria IDs present in the spec, and MUST NOT invent IDs.
- **FR-004**: The tasks-stage directive MUST instruct the agent to ensure the
  execution ledger exists for the feature after `tasks.md` is finalized, so that
  the implement stage can start tasks without a manual human command.
- **FR-005**: Each phase-bearing stage directive MUST instruct the agent to
  advance the ledger to the phase corresponding to that stage at the correct
  moment, using the SpecOps CLI (never by hand-editing the ledger).
- **FR-006**: The implement-stage directive MUST instruct the agent to open the
  review cycle when implementation completes, so `/specops-review` always finds
  an open cycle to record its outcome into.
- **FR-007**: All newly injected blocks MUST remain additive, idempotent
  (updated in place on re-run, never duplicated), and byte-identical-reversible
  on removal — identical to the guarantees the plan and implement blocks provide
  today.
- **FR-008**: Every injected stage directive MUST degrade gracefully: when
  SpecOps is not initialized or its CLI is unavailable, the underlying Speckit
  prompt MUST still complete its core work without a hard failure.
- **FR-009**: The existing plan-stage and implement-stage directive behavior
  MUST continue to work unchanged. If the `[SC-xxx]` rule is relocated or
  duplicated to the tasks stage, the plan and tasks guidance MUST NOT conflict.
- **FR-010**: All directive text MUST be in English (per the SpecOps language
  policy); it MUST parse only structural tokens and never client prose.
- **FR-011**: `specops init` MUST report each stage injection target and its
  status (installed / updated / unchanged), consistent with how it already
  reports plan and implement.
- **FR-012**: Ledger-mutating steps that the directives instruct MUST be
  expressed so that re-running a stage does not corrupt or duplicate ledger
  state — creation is guarded against an already-existing ledger, and phase
  advancement fails safe on an unexpected current phase.
- **FR-013**: The set of stages that receive phase/ledger wiring MUST be the
  phase-bearing linear path (specify, plan, tasks, implement, review). Auxiliary
  or read-only stages (clarify, analyze, checklist, converge, taskstoissues) are
  out of scope for phase transitions.

### Key Entities *(include if feature involves data)*

- **Stage directive block**: A marker-delimited unit of agent instructions
  injected into one Speckit stage prompt. New instances: specify and tasks.
  Existing: plan and implement.
- **Phase state machine**: The ordered ledger phases
  `SPECIFY → PLAN → TASKS → IMPLEMENT → REVIEW → DONE` and the allowed
  transitions between them, including review-cycle bookkeeping.
- **Execution ledger**: The per-feature `status.yaml` that holds task states,
  evidence, phase, and review cycles; created once per feature and mutated only
  through SpecOps commands.
- **Injection target set**: The per-integration mapping (resolved from Speckit
  manifests) of which prompt file each stage directive is written into.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An agent driving a fresh feature from specify through review
  completes the full phase progression with zero ledger or phase CLI commands
  issued manually by the human at stage seams.
- **SC-002**: In a SpecOps-initialized repo, the implement stage's first task
  transition never fails due to a missing ledger (0 such failures across the
  happy path).
- **SC-003**: 100% of task lines generated during the tasks stage carry a valid
  `[SC-xxx]` tag, and `specops consistency` reports 0 coverage violations on
  freshly generated tasks.
- **SC-004**: At every phase-bearing stage, the ledger phase reported by
  `status show` matches the stage the work is actually in.
- **SC-005**: Uninstalling SpecOps restores 100% of touched Speckit prompts to a
  byte-identical pre-injection state.
- **SC-006**: In a repo where SpecOps is not initialized, 100% of the injected
  Speckit stage prompts still complete their core work with no hard failure.
- **SC-007**: Re-running `specops init` after injection produces no duplicated
  blocks and leaves each prompt idempotent (unchanged content on a no-op
  re-run).

## Assumptions

- **Ledger creation point**: The ledger is created during the **tasks** stage,
  after `tasks.md` exists, because ledger creation syncs task IDs from
  `tasks.md`. The specify and plan stages therefore run before the ledger
  exists; their directives prepare for it and do not attempt ledger mutations
  that require a ledger. **Confirmed by the requester (2026-07-05)**: the ledger
  is created at the tasks stage; the SpecOps CLI requires no change to support
  this. *(Impacts US1, US2, FR-004, FR-005)*
- **Phase wiring scope**: Only the linear phase-bearing stages (specify, plan,
  tasks, implement, review) receive phase/ledger wiring. Clarify, analyze,
  checklist, converge, and taskstoissues are auxiliary and are left untouched.
- **Mechanism reuse**: This feature reuses the existing marker-delimited
  injection engine and manifest-driven target resolution; it does not introduce
  a new injection mechanism.
- **Single integration tested**: As today, the feature is verified against the
  Claude skills layout (separator `-`); other integrations are supported through
  the same manifest resolution but validated opportunistically.
- **No new runtime dependencies**: The change is expected to be additional
  template assets plus wiring in the initializer, requiring no new third-party
  dependencies.
- **Directive-as-instruction**: The directives instruct the agent to run
  existing SpecOps CLI commands; this feature does not require new CLI commands,
  only (possibly) making existing ones safe to re-invoke (FR-012).
