# Feature Specification: End-to-End Traceability

**Feature Branch**: `010-end-to-end-traceability`

**Created**: 2026-07-21

**Status**: Draft

**Input**: User description: "Build deterministic end-to-end traceability from success criteria through tasks, contexts and paths, commits, evidence, review findings, and corrections. Introduce planned, discovered-and-acknowledged, and unexplained path classes so review blocks unexplained drift without rejecting legitimate discoveries."

## Complement Boundary *(mandatory context)*

SpecOps runs **inside** Spec Kit and is a **complement**, never a replacement (Constitution Principle I). The prior features established the raw materials this feature stitches together: Feature 006 gave the versioned, concurrency-safe ledger that already records task records, commit hashes, evidence strings, and review cycles; Feature 009 wired the context map into planning so tasks and review records carry declared paths and context provenance (resolved context IDs + map digest). What has been missing is a single, deterministic, machine-checkable **trace** that connects each of a specification's success criteria forward through the tasks that satisfy it, the contexts and paths that task touched, the commits that landed it, the evidence that closed it, and the review findings and corrections that followed — and, running the other direction, that accounts for **every path in the effective diff** against that trace.

This feature adds exactly that trace layer plus the drift-classification it enables, contributing only:

1. A deterministic **trace model** linking success criteria → tasks → contexts/paths → commits → evidence → review findings → corrections, materialized from the Feature 006 ledger and Feature 009 provenance.
2. **Trace validation** that detects missing links, dangling references, and contradictory ownership, and fails closed with specific diagnostics.
3. A **path-class model** — `planned`, `discovered-and-acknowledged`, `unexplained` — computed over the **effective diff** (feature branch vs baseline).
4. A one-time **acknowledgement** action that moves a discovered path from `unexplained` to `discovered-and-acknowledged` with a concise reason and an associated task, recorded in the ledger.
5. A **review drift gate** that blocks review **only** on `unexplained` effective-diff paths, never on legitimate planned or acknowledged changes.
6. Human-readable and stable **JSON trace reports**.

Consistent with roadmap Rule 8 and the explicit non-goals below, this feature builds **no** Spec Kit workflow engine, gate, loop, or resume mechanism; it does **not** restructure review findings into first-class versioned state (that is Feature 011 — here findings and corrections are linked in their current form) and does **not** redesign the evidence/test-result artifact (that is Feature 012 — here evidence is linked as the existing `<CLASS>:<summary>` record). It composes what Features 006 and 009 already own and adds only the trace graph, its validation, the three path classes, the acknowledgement record, and the drift gate.

## Clarifications

### Session 2026-07-21

- Q: What per-task completeness rule does trace validation enforce, given Principle III's one-commit-per-user-story granularity? → A: Every task MUST carry evidence; **only the user-story-final task MUST additionally carry a commit**. Intermediate tasks are complete with evidence alone and MUST NOT be flagged for lacking a commit. This feature only *reads/links* the evidence the ledger already holds (a `<CLASS>:<summary>` string recorded at close via `--evidence` for intermediate tasks, or `--auto` for the final task) — it adds no new evidence collection or test execution cost.
- Q: When is a success criterion "completed" for the purpose of the trace gate? → A: **Mechanically, from ledger task state**: an SC is *completed* when **all** tasks tagged `[SC-xxx]` with it are complete in the ledger; an SC with any incomplete tagged task is *in progress* and not yet gated. Completion is never a separate human flag or agent judgment. Distinctly, an SC with **zero** covering tasks is an *uncovered-SC* missing-link defect regardless of completion state.
- Q: What does an acknowledgement bind to — does it stay valid if the path is later changed differently? → A: **Path-level** binding: an acknowledgement covers the path for the remainder of the feature; later changes to the same path (including after it leaves and re-enters the effective diff) remain `discovered-and-acknowledged` without re-acknowledgement. The map digest recorded with the acknowledgement is **provenance only** (for deterministic reproduction) and does **not** by itself invalidate the acknowledgement or revert the path to `unexplained`.
- Q: How are review findings and corrections linked into the trace while keeping structured finding IDs out of scope (Feature 011)? → A: Link each finding to a **path** via the `[File]` token already present in its `revisions/revision-X.md` line and to its **review cycle**; link corrections via the corrective **commits** recorded in that cycle's Feature 006 ledger record. **No** per-finding IDs are introduced — that anchoring is deferred to Feature 011.

### Session 2026-07-21 (post-checklist)

- Q: How is the effective-diff **baseline** resolved? → A: Use the **ledger-recorded baseline commit** (Feature 006) as the authoritative baseline; fall back to the **merge-base with the default branch** only when no ledger baseline exists (e.g. no-ledger/lightweight use). Explicit `--path` arguments still bypass Git derivation entirely (FR-002).
- Q: When a path qualifies as both `planned` and `discovered-and-acknowledged`, which class wins? → A: **Discovery wins** — if a genuine acknowledgement record exists, the path is classified `discovered-and-acknowledged` even when the plan later covers it, preserving the execution-time discovery provenance for review. This is safe because both classes pass the drift gate (only `unexplained` blocks), so precedence changes only the audit label, never the gate outcome. It stays coherent with FR-007: acknowledging an already-planned, never-discovered path remains a no-op, so a discovery label is never fabricated — it only wins when the discovery was real. The trace report MUST surface `discovered-and-acknowledged` paths distinctly with their reason and associated task so review scrutinizes discoveries specifically.
- Q: How are the new persisted ledger fields and the JSON output versioned? → A: **Both** — increment the Feature 006 ledger schema version for the new trace/acknowledgement fields with a forward-migration test, **and** embed an explicit schema-version field in every JSON output. Satisfies the Global Definition of Done ("persisted formats are versioned and have forward migration tests") and gives Features 011–014 a stable, versioned consumption surface consistent with how Features 006/008 version their formats.
- Q: How is concurrent acknowledgement handled (two sessions writing at once)? → A: **Reuse Feature 006's concurrency control** (lock / compare-and-swap): a concurrent or stale acknowledgement write MUST fail closed without lost updates, identical to every other ledger mutation. No new concurrency mechanism is introduced.
- Q: How are renames and mode-only changes counted in the effective diff? → A: **Decompose** — a rename counts as a removed old path plus an added new path, each classified independently, with **no** similarity-threshold rename detection (which would be non-reproducible); mode-only/permission changes with no content delta are still included as changed paths. Keeps every path independently explainable and deterministic (Features 008/009).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Block only unexplained diff drift at review (Priority: P1)

A reviewer runs the review drift gate on a feature branch. SpecOps computes the **effective diff** (the net set of changed paths of the feature branch relative to its baseline) and classifies each changed path as `planned` (declared in the plan or owned by a declared context per Feature 009), `discovered-and-acknowledged` (previously acknowledged with a reason and a task), or `unexplained` (in the diff but neither planned nor acknowledged). The gate passes (exit `0`) when every effective-diff path is `planned` or `discovered-and-acknowledged`, and fails closed (exit `1`) listing exactly the `unexplained` paths — so legitimate discovered work is never rejected while silent scope drift always is.

**Why this priority**: This is the core deliverable and the acceptance gate — reviews that block only on unexplained drift. It is independently valuable even before the full trace report exists, because it turns "did anything change that nobody planned or acknowledged?" into a deterministic, gate-shaped answer.

**Independent Test**: With a fixed branch, baseline, plan, and (optionally) context map, run the drift gate and confirm it (a) classifies every effective-diff path into exactly one of the three classes, (b) exits `0` when no path is `unexplained`, (c) exits `1` and names precisely the `unexplained` paths when any exist, and (d) never mutates repository or ledger state.

**Acceptance Scenarios**:

1. **Given** a feature branch whose every effective-diff path is declared in the plan or owned by a declared context, **When** the reviewer runs the drift gate, **Then** it reports success (exit `0`), classifies each path as `planned`, and lists zero `unexplained` paths.
2. **Given** a branch containing a changed path that is neither planned nor acknowledged, **When** the drift gate runs, **Then** it fails closed (exit `1`) and lists that path as `unexplained` without mutating any state.
3. **Given** a changed path previously recorded as `discovered-and-acknowledged`, **When** the drift gate runs and no other path is unexplained, **Then** it reports success (exit `0`) and classifies that path as `discovered-and-acknowledged`, not `unexplained`.
4. **Given** a clean working tree / empty effective diff, **When** the drift gate runs, **Then** it reports the empty-diff state as a supported non-error (exit `0`) with zero paths in every class.
5. **Given** the same branch, baseline, plan, map, and acknowledgements, **When** the gate is run repeatedly, **Then** the per-path classification and its ordering are byte-for-byte identical across runs.

---

### User Story 2 - Acknowledge a legitimate discovery once (Priority: P2)

While implementing, a maintainer discovers that a file not predicted during planning must change (a genuine, in-scope discovery). Rather than being blocked at review, the maintainer acknowledges the path once, attaching a concise reason and the task the discovery belongs to. SpecOps records the acknowledgement in the ledger so the drift gate thereafter classifies that path as `discovered-and-acknowledged`. A second acknowledgement of the same path is idempotent when identical and fails closed when it conflicts, so the audit record is never silently overwritten.

**Why this priority**: Acknowledgement is what makes the P1 gate humane — without it every discovery is `unexplained`. It is a distinct, independently testable state-changing slice that depends only on the class model and the ledger.

**Independent Test**: Acknowledge a discovered path with a reason and task, then run the drift gate and confirm the path is now `discovered-and-acknowledged`; re-run the identical acknowledgement and confirm idempotency; attempt a conflicting acknowledgement (different task or reason) and confirm it fails closed without altering the existing record.

**Acceptance Scenarios**:

1. **Given** an `unexplained` effective-diff path and a valid task ID, **When** the maintainer acknowledges the path with a reason and that task, **Then** the acknowledgement is written atomically to the ledger and the path is subsequently classified `discovered-and-acknowledged`.
2. **Given** an already-acknowledged path, **When** the identical acknowledgement (same path, task, and reason) is issued again, **Then** the operation is idempotent (exit `0`) and produces no duplicate record.
3. **Given** an already-acknowledged path, **When** a conflicting acknowledgement (different task or different reason) is issued, **Then** the command fails closed (exit `2`) and leaves the existing acknowledgement unchanged.
4. **Given** a task ID that does not exist in the ledger, **When** an acknowledgement references it, **Then** the command fails closed (exit `2`) and records nothing (no dangling task reference is ever created).
5. **Given** a path that is already `planned`, **When** the maintainer attempts to acknowledge it, **Then** the command reports it as already-explained (exit `0`, no-op) rather than creating a redundant acknowledgement.

---

### User Story 3 - Report and validate the end-to-end trace (Priority: P2)

A reviewer or auditor asks SpecOps for the complete trace of the feature: for each success criterion, the tasks that satisfy it, the contexts and paths those tasks touched, the commits that landed them, the evidence that closed them, and the review findings and corrections that followed. SpecOps materializes this trace deterministically from the ledger and emits both a concise human-readable report and a stable JSON contract. A companion validation pass fails closed when the trace is incomplete — a completed success criterion with no covering task, a task with no commit or evidence, a dangling reference to a missing commit/task/finding, or contradictory path ownership — producing a specific diagnostic for each gap.

**Why this priority**: The trace report and its validation are the auditability payload of the milestone and satisfy the "every completed success criterion has a complete machine-checkable trace or an explicit blocking diagnostic" half of the acceptance gate. They build on the same ledger data as P1/P2 and are independently testable against fixtures.

**Independent Test**: Against a fixture ledger with a known-complete trace, run the report and confirm every success criterion resolves to its full chain and the validation exits `0`; against fixtures seeded with a missing link, a dangling reference, and a contradictory ownership case, confirm validation exits `1` with one specific diagnostic per defect and that the JSON output is stable and byte-for-byte reproducible.

**Acceptance Scenarios**:

1. **Given** a ledger whose every completed success criterion has tasks, commits, and evidence, **When** the reviewer runs trace validation, **Then** it reports success (exit `0`) and the report renders each success criterion's full chain to corrections.
2. **Given** a success criterion marked complete but covered by no task, **When** validation runs, **Then** it fails closed (exit `1`) and names the uncovered success criterion as a missing-link defect.
3. **Given** a ledger record referencing a commit hash absent from the active branch's Git tree, **When** validation runs, **Then** it fails closed (exit `1`) and reports the dangling reference (deferring commit-existence enforcement to `specops reconcile` while surfacing the trace-level break).
4. **Given** a changed path a task associates with one context while the map owns it under a different, undeclared context, **When** validation runs, **Then** it fails closed (exit `1`) and reports the contradictory ownership with both contexts named.
5. **Given** identical ledger, branch, and map inputs, **When** the JSON trace report is produced repeatedly, **Then** its content and ordering are byte-for-byte identical across runs.

---

### Edge Cases

- **No context map present**: path classification falls back to plan-declared paths only (Feature 009's supported no-map state); `planned` means "declared in the plan," ownership contradictions cannot arise, and no context requirement is imposed.
- **Not a Git repository or no resolvable baseline**: the effective diff cannot be derived; the drift gate and any Git-deriving command fail as a usage/input error (exit `2`) with a clear message, never as a silent empty result.
- **Pre-feature ledger without trace/acknowledgement fields**: treated as a valid supported prior shape via additive forward migration; reads never fail and absent fields are reported explicitly, not as defects.
- **Acknowledged path later leaves the diff**: the acknowledgement remains recorded but simply goes unused by classification; it is never resurfaced as a defect and re-appears correctly if the path is changed again.
- **A completed success criterion whose tasks exist but whose commits/evidence are absent**: reported as a missing-link defect, distinct from the "no task at all" defect, so the exact break in the chain is identifiable.
- **Effective diff includes a path owned by a context declared for a *different* task than the one that changed it**: reported as contradictory ownership rather than silently accepted.
- **SpecOps/Speckit-managed artifact paths** (the **active feature's** `specs/<feature>/**`, `.specify/**`, `specops.json`): excluded from effective-diff classification — they are methodology state (e.g. the ledger's `status.yaml` changes on every task close), not product drift; classifying them would false-block review (SC-003). Only the active feature directory under `specs/` is excluded, so a repository that stores product artifacts under a top-level `specs/` is still classified.
- **No classification basis** (no plan path declarations, no context map, and no acknowledgements): the always-on **review drift gate degrades to a SKIPPED gate** (never a block) — every path would otherwise be `unexplained`, which is not a meaningful drift signal. This keeps upgrading a repository whose plan predates the path-declaration convention from being retroactively rejected (roadmap Rule 5). The explicit `specops trace classify`/`validate` commands remain strict.
- **Effective diff cannot be derived** (not a Git repo, or no resolvable baseline): the drift gate and `trace validate` **fail closed** — the gate FAILs and `validate` returns the usage error (exit `2`); neither reports a clean pass for drift that was never evaluated (Principle VI).
- **Deleted (removed) path in the effective diff**: classified like any other changed path; a removal that was neither planned nor acknowledged is `unexplained`.
- **Renamed path**: decomposed into a removed old path and an added new path, each classified independently (no similarity-threshold rename detection); the new path can be `unexplained` on its own if neither planned nor acknowledged.
- **Path both planned and acknowledged**: classified `discovered-and-acknowledged` (discovery precedence, FR-003), preserving discovery provenance; the drift-gate outcome is unaffected since both classes pass.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: SpecOps MUST materialize a deterministic **trace** that links each specification success criterion to the task(s) that satisfy it (via the `[SC-xxx]` task tags), and each task forward to its declared contexts/paths, its commits, its evidence record, and any associated review findings and corrections, composing the Feature 006 ledger and Feature 009 provenance without re-deriving that state from agent narration. A success criterion is **completed** when, and only when, every task tagged with it is complete in the ledger; completion MUST be derived mechanically from ledger task state, never from a separate human flag or agent judgment.
- **FR-002**: SpecOps MUST compute the set of changed paths as the **effective diff** of the active feature branch relative to its baseline (the net changed paths, not intermediate churn), **refining** Feature 009's `context impact` Git derivation (it MUST additionally disable rename detection — see below — which `context impact` does not); explicit path arguments, when supplied, MUST bypass Git derivation for hermetic testing. The **baseline** MUST resolve to the ledger-recorded baseline commit (Feature 006) when present, falling back to the merge-base with the default branch only when no ledger baseline exists; the resolved baseline reference MUST be reported in output so classification is reproducible and its origin auditable. A **rename** MUST be decomposed into a removed old path plus an added new path, each classified independently, with **no** similarity-threshold rename detection; **mode-only/permission** changes with no content delta MUST still be counted as changed paths. A **symlink** MUST be matched by its own path entry and MUST NOT be followed (mirroring Feature 009 stale-detection). This keeps the change set deterministic and every path independently explainable.
- **FR-003**: SpecOps MUST classify **every** effective-diff path into exactly one of three classes: **`planned`** (declared in the plan, or — when a map is present — owned by a context the plan declared), **`discovered-and-acknowledged`** (covered by a recorded acknowledgement), or **`unexplained`** (in the diff but neither planned nor acknowledged). Every path MUST receive exactly one class and the assignment MUST be attributable in the output. When a path qualifies as both `planned` and `discovered-and-acknowledged` (its acknowledgement predates a later plan amendment), **`discovered-and-acknowledged` takes precedence** so the execution-time discovery provenance is preserved; this is safe because both classes pass the drift gate and precedence changes only the audit label. Combined with FR-007's no-op guard, the discovery label is never fabricated for a path that was planned from the start and never genuinely discovered.
- **FR-004**: SpecOps MUST provide a read-only **review drift gate** that exits `0` when no effective-diff path is `unexplained` and exits `1`, listing precisely the `unexplained` paths, when any exist. The gate MUST NOT block on `planned` or `discovered-and-acknowledged` paths, and MUST NOT mutate repository or ledger state.
- **FR-005**: SpecOps MUST allow a discovered path to be **acknowledged once** with a concise human-supplied reason and an associated task ID, recording the acknowledgement in the Feature 006 ledger atomically and interruption-safely. Acknowledgement binding is **path-level**: once recorded, the acknowledgement classifies that path as `discovered-and-acknowledged` for the remainder of the feature — including after the path leaves and re-enters the effective diff, and regardless of how the path's content later changes — and MUST NOT require re-acknowledgement. The map digest stored with the record (FR-017) is provenance for deterministic reproduction only and MUST NOT, by itself, invalidate the acknowledgement. Concurrent or stale acknowledgement writes MUST be guarded by Feature 006's existing concurrency control (lock / compare-and-swap) and fail closed without lost updates, identically to other ledger mutations — no new concurrency mechanism is introduced.
- **FR-006**: Acknowledgement MUST be idempotent for an identical (path, task, reason) triple and MUST fail closed (exit `2`) on a conflicting acknowledgement (same path, different task or reason), never silently overwriting an existing acknowledgement, so the audit record is preserved.
- **FR-007**: Acknowledgement MUST reject a non-existent task ID (exit `2`) and record nothing on rejection, so no acknowledgement can create a dangling task reference. Acknowledging a path that is already `planned` MUST be reported as already-explained (exit `0`, no-op) rather than creating a redundant record.
- **FR-008**: SpecOps MUST NOT hard-reject a changed path solely because it was not predicted during planning; a discovered path becomes non-blocking through acknowledgement (FR-005), and only genuinely `unexplained` paths block review (FR-004). This preserves Feature 009's guarantee that unpredicted discoveries are not, by themselves, blocking.
- **FR-009**: SpecOps MUST provide read-only **trace validation** that fails closed (exit `1`) on: (a) a success criterion with **zero** covering tasks (**uncovered-SC missing link**), independent of completion state, (b) a completed task with **no evidence**, or a **user-story-final** task with no commit (**missing link**) — intermediate tasks, which by Constitution Principle III close with evidence but no commit, MUST NOT be flagged for lacking a commit, (c) a ledger reference to a commit, task, finding, or acknowledgement that cannot be resolved (**dangling reference**), and (d) a path a task associates with one context while the map owns it under a different, undeclared context (**contradictory ownership**). Each defect MUST be reported as a distinct, specific diagnostic.
- **FR-010**: For commit references, trace validation MUST surface the trace-level break as a dangling reference but MUST defer authoritative commit-existence enforcement to `specops reconcile` (Constitution Principle II), rather than duplicating or contradicting that check.
- **FR-011**: SpecOps MUST provide a **trace report** in both a concise human-readable form and a stable JSON contract, each rendering the full chain from success criteria through tasks, contexts/paths, commits, evidence, findings, and corrections, suitable for audit and for consumption by later features. The report MUST surface `discovered-and-acknowledged` paths **distinctly** — each with its recorded reason and associated task — so a reviewer can scrutinize execution-time discoveries specifically rather than having them absorbed into the `planned` set.
- **FR-012**: All trace read commands (classification, drift gate, report, validation) MUST be **read-only**: they MUST NOT mutate repository or ledger state, verifiable by before/after state comparison. Only acknowledgement (FR-005) changes state.
- **FR-013**: Every trace command MUST treat a **missing context map** as a supported, explicitly reported state: classification falls back to plan-declared paths, ownership-contradiction checks are inapplicable, and no context requirement is imposed (roadmap Rule 5).
- **FR-014**: Every read-only trace command MUST expose the fixed exit-code taxonomy already established for SpecOps context commands — `0` success (including supported empty-diff, no-map, and all-explained states), `1` blocking/fail-closed (any `unexplained` diff path; any missing link, dangling reference, or contradictory ownership), `2` usage/input error (not a Git repository, no resolvable baseline, conflicting or invalid acknowledgement, bad arguments) — with the fine-grained outcome carried in a stable `status` field. Every JSON output MUST additionally embed an explicit **schema-version** field so consumers (Features 011–014) can bind to a versioned contract.
- **FR-015**: For identical ledger, branch, map, and acknowledgement inputs, every trace command MUST produce **byte-for-byte identical** output (deterministic ordering and serialization), consistent with the determinism guarantees of Features 008 and 009.
- **FR-016**: SpecOps MUST preserve read compatibility with ledgers written before this feature; task/review records lacking trace or acknowledgement fields MUST be treated as a valid supported prior shape via additive forward migration and MUST NOT block reads or be reported as defects.
- **FR-017**: The acknowledgement record and any new trace fields MUST be a **versioned, additive** extension of the ledger written through the Feature 006 CLI surface (never hand-edited or held in agent memory), and MUST carry enough information (path, task, reason, and the map digest in effect when a map is present) for a later session to reproduce the classification deterministically. Adding these fields MUST **increment the Feature 006 ledger schema version** and be covered by a forward-migration test that upgrades a pre-feature ledger without data loss.
- **FR-018**: SpecOps MUST NOT restructure review findings into first-class versioned state (deferred to Feature 011) and MUST NOT redesign the evidence/test-result artifact (deferred to Feature 012); this feature links findings and corrections in their current form and links evidence as the existing `<CLASS>:<summary>` record. Specifically, each finding MUST be linked to a path via the `[File]` token in its `revisions/revision-X.md` line and to its originating review cycle, and corrections MUST be linked via the corrective commits recorded in that cycle's ledger record — **without** introducing any per-finding identifier (that anchoring is Feature 011).
- **FR-019**: SpecOps MUST NOT introduce a language- or framework-specific source-code dependency or diff parser; path classification and trace links MUST derive only from the Git effective diff, the plan's declared paths, the Feature 008/009 context map and provenance, and the Feature 006 ledger, keeping the core stack-neutral (Constitution Principle V).

### Key Entities *(include if feature involves data)*

- **Trace Graph**: The deterministic linkage from each success criterion → covering task(s) → declared contexts and paths → commits → evidence → review findings → corrections, materialized from the ledger and provenance; the backbone all trace commands render or validate.
- **Effective-Diff Path**: A single path in the net change set of the feature branch versus its baseline, carrying exactly one path class and the attribution for that class.
- **Path Class**: One of `planned`, `discovered-and-acknowledged`, or `unexplained` — the closed classification that determines whether a changed path blocks review.
- **Acknowledgement Record**: A one-time, ledger-stored assertion that a discovered path is legitimate, carrying the path, a concise reason, the associated task ID, and the map digest in effect; the sole state-changing artifact this feature introduces.
- **Trace Defect**: A specific, machine-checkable finding produced by validation — a missing link, a dangling reference, or a contradictory ownership — each with enough locating detail to act on.
- **Trace Report**: The human-readable and JSON renderings of the Trace Graph plus the path classification for the current effective diff.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For any fixed branch, baseline, plan, map, and acknowledgement set, path classification and every trace report are 100% reproducible — repeated runs produce byte-for-byte identical results, including per-path attribution and ordering.
- **SC-002**: 100% of effective-diff paths are assigned exactly one class from the closed set {`planned`, `discovered-and-acknowledged`, `unexplained`}, with zero paths unclassified and zero paths in more than one class, across all test fixtures.
- **SC-003**: The review drift gate blocks (exit `1`) in 100% of fixtures containing at least one `unexplained` path and passes (exit `0`) in 100% of fixtures whose every path is `planned` or `discovered-and-acknowledged`; it never blocks a `planned` or `discovered-and-acknowledged` path (0% false-block rate).
- **SC-004**: 100% of legitimate discoveries acknowledged once with a reason and a valid task are reclassified from `unexplained` to `discovered-and-acknowledged`; identical re-acknowledgement is idempotent (no duplicate records) and conflicting or invalid-task acknowledgement fails closed (exit `2`) with the prior record unchanged, in 100% of fixtures.
- **SC-005**: 100% of seeded trace defects — a completed success criterion with no covering task, a task missing a required commit or evidence, a dangling commit/task/finding/acknowledgement reference, and a contradictory path ownership — are detected and reported as distinct diagnostics at exit `1`, with zero false positives on a known-complete trace.
- **SC-006**: Every completed success criterion in a known-complete fixture resolves, in the trace report, to a full chain (task → contexts/paths → commit → evidence → findings/corrections) with no broken segment; the acceptance-gate property "complete machine-checkable trace or explicit blocking diagnostic" holds for 100% of completed success criteria and 100% of effective-diff paths.
- **SC-007**: Every read-only trace command returns within the fixed `0`/`1`/`2` exit-code taxonomy with a populated `status` field and never mutates repository or ledger state (verified by before/after comparison); pre-feature ledgers remain readable in 100% of migration tests.
- **SC-008**: When no context map is present, 0% of paths are blocked for context-ownership reasons and classification uses plan-declared paths only; when a map is present, 100% of contradictory-ownership fixtures are detected — with the same command surface behaving correctly in both modes.

## Assumptions

- The versioned, concurrency-safe ledger from Feature 006 is the storage for acknowledgements and any new trace fields; these are additive, forward-migrated extensions written exclusively through the SpecOps CLI (Constitution Principle II), preserving read compatibility with prior ledger shapes.
- Success-criterion → task coverage is already asserted at plan time by `specops consistency` (Constitution Principle IV); this feature **extends** that coverage into a full runtime chain (task → path/context → commit → evidence → finding → correction) validated at review time, and complements rather than duplicates the consistency check.
- The "effective diff" and its Git derivation (active branch vs baseline, with explicit-path override) **refine** Feature 009's `context impact` derivation — adding `--no-renames` rename decomposition and symlink-by-path matching — rather than reusing it verbatim; computing the diff is an implementation detail, not new schema.
- "Planned" paths are the plan's declared paths (carrying Feature 004/009 action suffixes) and, when a map is present, the paths owned by declared contexts per Feature 009; ownership is an explanatory/topology relationship, never an exclusive write-permission boundary.
- Context provenance (resolved context IDs + map digest) required to detect contradictory ownership is the provenance Feature 009 already snapshots into task/review records; this feature reads it and does not redefine how it is computed.
- Review findings and corrections are linked in their **current** form (the `revisions/revision-X.md` short-format findings and their corrective commits); their promotion to first-class versioned state with lifecycle transitions is Feature 011 and is out of scope here.
- Evidence is linked as the existing machine-collected `<CLASS>:<summary>` record (Constitution Principle III); the structured-evidence and gate-profile redesign is Feature 012 and is out of scope here.
- Commit-existence enforcement remains owned by `specops reconcile` (Constitution Principle II); trace validation surfaces a trace-level dangling reference but defers the authoritative block to reconcile to avoid duplicated or contradictory gating.
- English and Portuguese documentation remain behaviorally equivalent, and every new surface degrades safely to a supported state when no context map exists (roadmap Rule 5).
