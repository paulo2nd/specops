# Feature Specification: Gate Profiles and Structured Evidence

**Feature Branch**: `012-gate-profiles-evidence`

**Created**: 2026-07-23

**Status**: Draft

**Input**: User description: "Add ordered, context-aware gate profiles and migrate evidence from summary strings to versioned records with command, result, timestamp, commit range, paths, provenance, and digests. Provide stable JSON reporting, safe caching, migration compatibility, and optional SARIF output."

## Complement Boundary *(mandatory context)*

SpecOps runs **inside** Spec Kit and is a **complement**, never a replacement (Constitution Principle I). Today the verification the review agent runs is a **single pair of global commands** — `test_command` and `lint_command` from `specops.json` — evaluated by a fixed cheapest-first gate list (`reconcile → lint → test → working-tree → drift`) where each command gate is all-or-nothing: it runs one command for the whole repository, regardless of which contexts (Feature 008) or paths actually changed, and its outcome is a bare `PASS`/`FAIL`/`SKIPPED`. The evidence that closing a task records is likewise a **flat summary string** in the `<CLASS>:<summary>[; …]` grammar (`TEST_REPORT:…; CODE_DIFF:…`, classes `CLI_LOG | TEST_REPORT | SCREENSHOT_PATH | CODE_DIFF`): human-readable, but opaque to a machine — it does not record *which* command produced it, its exit code, *when* it ran, over *which* commit range and paths, or a digest that lets a later step decide the result is still valid. A review verdict therefore cannot answer, from repository state alone, "exactly which gates ran, why they applied, what inputs they covered, and which immutable evidence supports this result."

This feature promotes verification configuration and evidence to first-class, versioned state, contributing only:

1. **Ordered gate profiles** — a versioned, stack-neutral definition of named gates, each carrying its command, an applicability predicate (context / risk / changed-path scope), a timeout, a required-status classification (required vs optional), and its failure semantics — replacing the single global `test_command`/`lint_command` pair with a deterministically selected, ordered suite.
2. **Deterministic profile selection** — resolving which gates apply to a given run from explicit configuration, the resolved contexts (Feature 008), declared risk, and the changed paths, with an explainable reason for every gate that ran and every gate that was skipped.
3. **Structured evidence records** — a versioned evidence object carrying producer, command, exit code, timestamp, commit range, affected paths, a summary, and an optional artifact digest — the machine-checkable successor to the `<CLASS>:<summary>` string, linked wherever a task or finding links evidence today.
4. A **gate-outcome taxonomy** that distinguishes `required`, `optional`, `skipped`, `cached`, `failed`, and `unavailable`, so a verdict states not just pass/fail but the disposition of every gate.
5. **Safe evidence caching** — reuse of a prior evidence record only when its command, inputs, context-map digest, and commit all still match, so a cached result is never silently trusted across a change that could invalidate it.
6. **Stable JSON reporting and an optional SARIF output adapter** — a versioned JSON contract over profiles, selection reasons, gate outcomes, and evidence, plus an opt-in SARIF projection of review findings for external tooling.

Consistent with roadmap Rule 8 and the non-goals below, this feature builds **no** Spec Kit workflow engine, human gate, loop, or resume mechanism. A SpecOps gate **profile** is a deterministic verification-command suite invoked as native Spec Kit `shell`/`command` steps that produces evidence — a **distinct concept** from Spec Kit's human `gate` step and its capability system, which this feature neither overlaps nor replaces. SpecOps does **not** interpret test-framework-specific result formats in the core (an exit code and a captured summary are the contract; parsing a framework's XML/JSON is out of scope), stores **no** remote artifacts (digests reference local content only), and does not redesign the finding lifecycle (Feature 011). It composes Feature 006's versioned, concurrency-safe ledger and migration machinery, Feature 008's context map and its digest, Feature 009's impact/changed-path resolution, and Feature 010/011's evidence-linking points — adding only the gate-profile schema, the selection logic, the structured evidence object, the outcome taxonomy, the caching key, and the JSON/SARIF reports.

## Clarifications

### Session 2026-07-23

- Q: How should a gate's applicability be determined, given Feature 008's context map already ships a per-context `gates` list and a free-form `risk` mapping? → A: **Self-declared, map-aware**. Each gate declares **one** applicability predicate in the profile config combining any of `always` (global), context ids, changed-path globs, and a risk condition; the predicate MAY reference map contexts by id, and a resolved context's own `gates` list is honored as an **implicit context-id match**. One selection language that works with or without a map (roadmap Rule 5), composing the field Feature 008 already ships rather than adding a second, parallel selector.
- Q: Where does the gate-profile suite execute, relative to the existing `specops review` gate pipeline? → A: **Integrated into review**. The selected profile suite **replaces** the fixed `lint`/`test` gates inside `specops review` (pipeline becomes `reconcile → [selected profile suite] → working-tree → drift`); selection, execution, and evidence become part of the existing review verdict. Read-only `specops gate list`/`validate`/`report` surfaces expose selection and evidence; **no** separate standalone gate runner is introduced. This is the minimal composition of the shipped review surface and precedes the Feature 017 `preflight` rename.
- Q: How does a gate's risk-based applicability match Feature 008's free-form `risk` mapping? → A: **Named-key presence/equality**. A gate's risk condition is satisfied when a resolved context's free-form `risk` mapping contains a specified key (optionally equal to a declared value). SpecOps imposes **no** ordinal risk scale and does not otherwise interpret the mapping (Constitution Principle V); the selection reason is `matched risk key`.
- Q: Do structured evidence records carry a stable identifier, and how is it derived? → A: **Cache-key-derived id**. Each evidence record's id is derived deterministically from its cache key (producer / command / commit range / affected paths / context-map digest), so identical production yields an identical id and a changed cache key yields a **new** record — under a **new** id — that supersedes (never mutates) the prior one. Verdicts and Feature 011 findings reference evidence by this id.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Define and deterministically select ordered gate profiles (Priority: P1)

A repository owner declares an **ordered set of gate profiles** in versioned SpecOps configuration: each profile names a gate, the command it runs, *where it applies* (a context id, a risk level, and/or a changed-path pattern), a timeout, whether it is **required** or **optional**, and what a non-zero exit means. When a run needs verification, SpecOps **selects** the applicable gates deterministically from that configuration together with the resolved contexts (Feature 008), the declared risk, and the changed paths (Feature 009), runs the required ones in declared order, and — for every gate — records **why it applied or was skipped**. A repository that declares no profiles keeps working: the legacy global `test_command`/`lint_command` degrade to an implicit default profile, so upgrading never breaks verification.

**Why this priority**: This is the core of the feature and of the acceptance gate — "every review verdict identifies exactly which gates ran and why they applied." It is independently valuable even before evidence is restructured, because it turns one blunt global command into a proportional, explainable, ordered suite. It is the foundation the evidence, taxonomy, and reporting slices build on.

**Independent Test**: From a fixture declaring several profiles scoped by context, risk, and changed-path pattern, resolve the applicable gates for a given changed-file set and confirm the selected gates, their order, and the applied/skipped reason for each are exactly as declared and byte-for-byte reproducible; from a fixture with **no** profile declaration but a legacy `test_command`/`lint_command`, confirm the default profile is synthesized and verification still runs.

**Acceptance Scenarios**:

1. **Given** an ordered profile set where gate B is scoped to a context and changed-path pattern that the current run matches and gate C is scoped to one it does not, **When** selection runs, **Then** B is selected and C is skipped, each with a specific applied/skipped reason, and required gates run in the declared order.
2. **Given** identical configuration, contexts, risk, and changed paths, **When** selection runs repeatedly, **Then** the selected set, order, and reasons are byte-for-byte identical across runs.
3. **Given** a repository with no gate-profile declaration but a legacy `test_command` (and optional `lint_command`), **When** selection runs, **Then** SpecOps synthesizes an implicit default profile from those commands and verification proceeds unchanged (roadmap Rule 5 degrade).
4. **Given** a gate-profile configuration with a duplicate gate name, a cyclic ordering, an unknown context reference, or an unparseable applicability predicate, **When** the configuration is validated, **Then** validation fails closed (exit `1`) with a specific diagnostic and no gate is run.
5. **Given** a `required` gate that exits non-zero and an `optional` gate that exits non-zero, **When** the suite runs, **Then** the required failure fails the verdict (fail-closed) while the optional failure is recorded but does not block, each per its declared failure semantics.

### User Story 2 - Record structured, migratable evidence (Priority: P2)

Whenever a gate runs — or a task or finding links evidence — SpecOps records a **structured evidence object** instead of a flat summary string: the producer (gate/command identity and version), the exact command, its exit code, a timestamp, the commit range it covered, the affected paths, a concise summary, and an optional digest of a captured artifact. The evidence lives in the versioned ledger; upgrading a repository whose ledger holds legacy `<CLASS>:<summary>` strings migrates those strings into structured records without loss, and reads of the prior shape never fail.

**Why this priority**: Structured evidence is what makes the acceptance gate's "which immutable evidence records support the result" answerable, and it is the substrate the outcome taxonomy and caching depend on. It is a distinct, independently testable slice — it can be exercised over recorded evidence without the full selection logic — but it consumes the run identity that US1 establishes, so it is P2.

**Independent Test**: Run a gate over a fixture and confirm the produced evidence object carries producer, command, exit code, timestamp, commit range, affected paths, summary, and (when an artifact is captured) a digest, all round-tripping byte-for-byte; take a ledger fixture holding legacy `<CLASS>:<summary>` evidence strings, run the forward migration, and confirm every legacy string becomes a structured record with zero loss and that the pre-migration ledger remains readable.

**Acceptance Scenarios**:

1. **Given** a gate command that exits `0` over a known commit range and path set, **When** the evidence is recorded, **Then** the structured object captures producer, command, exit code `0`, a timestamp, that commit range, those affected paths, a summary, and (if an artifact was captured) its digest — written atomically through the Feature 006 CLI.
2. **Given** a ledger holding legacy `<CLASS>:<summary>` evidence strings, **When** the forward migration runs, **Then** each string is converted to a structured evidence record preserving its class and summary, the ledger schema version is incremented, and no data is lost.
3. **Given** a pre-feature ledger, **When** any read command runs, **Then** records lacking structured-evidence fields read without error and absent fields are reported explicitly, never as defects.
4. **Given** identical inputs, **When** evidence is recorded and re-read, **Then** the serialized evidence object is byte-for-byte identical across reads.
5. **Given** an artifact digest recorded for a captured output, **When** the referenced local content is unchanged, **Then** the digest re-computes identically; **When** it changed, **Then** the mismatch is detectable (no remote storage is involved).

### User Story 3 - Classify gate outcomes and make the verdict fully provenanced (Priority: P2)

A reviewer or auditor asks what a verdict actually covered. SpecOps reports, for every gate in the selected suite, one of a fixed **outcome taxonomy** — `required`, `optional`, `skipped`, `cached`, `failed`, or `unavailable` — together with the reason it applied, the inputs it covered (commit range and paths), and the immutable evidence record that supports it. A gate whose result can be **safely reused** is marked `cached` only when its command, inputs, context-map digest, and commit all still match the prior evidence; any mismatch forces a fresh run rather than trusting stale evidence. A gate whose command is missing or whose tool is absent is `unavailable`, distinct from `failed`.

**Why this priority**: This closes the acceptance gate — "identifies exactly which gates ran, why they applied, what inputs they covered, and which evidence supports the result" — and delivers safe caching. It builds directly on US1's selection and US2's evidence, so it is P2 and independently testable over seeded selection + evidence fixtures.

**Independent Test**: From a fixture with a selected suite and seeded evidence, produce the verdict report and confirm every gate carries exactly one taxonomy value, its applied reason, its covered commit range/paths, and its supporting evidence record; seed a prior evidence record and re-run with (a) all cache-key components unchanged → the gate is `cached`, (b) a changed commit, (c) a changed context-map digest, (d) changed inputs → each forces a fresh run (not `cached`); seed a missing command → the gate is `unavailable`, not `failed`.

**Acceptance Scenarios**:

1. **Given** a completed suite, **When** the verdict is reported, **Then** each gate shows exactly one taxonomy value (`required` | `optional` | `skipped` | `cached` | `failed` | `unavailable`), its applied/skipped reason, its covered commit range and paths, and its supporting evidence record (where one exists).
2. **Given** a prior evidence record whose command, inputs, context-map digest, and commit all still match, **When** the gate is evaluated, **Then** it is reported `cached` and reuses that immutable evidence without re-running the command.
3. **Given** a prior evidence record where any one of command, inputs, context-map digest, or commit has changed, **When** the gate is evaluated, **Then** the cache is **not** reused, the gate runs fresh, and the new evidence supersedes (never mutates) the prior record.
4. **Given** a required gate whose command or tool is absent, **When** the gate is evaluated, **Then** it is reported `unavailable` (distinct from `failed`) and the verdict fails closed unless the profile declares the gate optional.
5. **Given** a required gate that exceeds its declared timeout, **When** it is evaluated, **Then** it is reported `failed` with a timeout reason and the verdict fails closed.

---

### User Story 4 - Emit stable JSON and optional SARIF for external tooling (Priority: P3)

An automation consumer (a CI job, or a downstream Feature 013/014 surface) needs the profiles, selection reasons, gate outcomes, and evidence as a stable machine contract, and an external code-review tool wants review findings in a standard format. SpecOps emits a **versioned JSON** report over the whole verification result and, opt-in, a **SARIF** projection of the review findings (Feature 011), each deterministic and byte-for-byte reproducible from identical state.

**Why this priority**: The JSON contract and SARIF adapter are the interoperability payload — valuable for CI and downstream features but not required to prove the core acceptance gate — so they are P3. They are independently testable by asserting schema-versioned, reproducible output against fixtures.

**Independent Test**: Against a fixture verification result, emit the JSON report and confirm it carries an explicit `output_version` field, renders every profile/selection-reason/gate-outcome/evidence record, and is byte-for-byte reproducible; enable the SARIF adapter over a findings fixture and confirm it emits schema-valid SARIF that round-trips the findings' rule, location, and severity, and that SARIF output is strictly opt-in (absent by default).

**Acceptance Scenarios**:

1. **Given** a completed verification result, **When** the JSON report is produced, **Then** it embeds an explicit `output_version` field and renders profiles, per-gate selection reasons, outcome-taxonomy values, covered inputs, and supporting evidence records.
2. **Given** identical ledger, configuration, and branch inputs, **When** the JSON report is produced repeatedly, **Then** its content and ordering are byte-for-byte identical.
3. **Given** review findings (Feature 011) and the SARIF adapter enabled, **When** SARIF is emitted, **Then** it is schema-valid and preserves each finding's rule, location, and severity mapping.
4. **Given** default configuration, **When** any report runs, **Then** SARIF is **not** emitted unless explicitly requested (opt-in), and its absence is never a defect.

### Edge Cases

- **No gate-profile configuration**: the legacy global `test_command`/`lint_command` are synthesized into an implicit default profile; verification runs unchanged and the repository is never blocked for lacking profiles (roadmap Rule 5).
- **No context map present**: context-id and risk-key predicates match nothing; `always` and changed-path predicates still select, so verification still runs. A missing map is a supported reported state (Feature 008), never a defect.
- **Empty changed-path set** (e.g. a no-op diff): path-scoped gates are `skipped` with an explicit reason; globally-scoped required gates still run.
- **Optional gate fails**: recorded as `failed` in its evidence but does not block the verdict, per its declared failure semantics; a required gate failing does block.
- **Gate command absent / tool not installed**: reported `unavailable`, distinct from `failed`; a required `unavailable` gate fails the verdict closed unless the profile marks it optional.
- **Gate exceeds its timeout**: terminated and reported `failed` with a timeout reason (fail-closed for required gates).
- **Cache-key partial match**: if command, inputs, context-map digest, and commit do not *all* match, the result is **not** cached — the gate runs fresh; a cached result is never trusted across any invalidating change.
- **Artifact digest references content that changed**: the digest mismatch is detectable; SpecOps stores no remote copy, so the evidence records the digest that was current when produced.
- **Legacy evidence strings in an upgraded ledger**: forward-migrated into structured records without loss; the pre-migration ledger remains readable and is never reported as a defect.
- **Framework-specific result files** (JUnit XML, coverage JSON): SpecOps records the command, exit code, and a captured summary only; it does **not** parse or interpret the framework format (core stays stack-neutral).
- **Two sessions recording evidence for the same gate concurrently**: serialized by Feature 006's concurrency control; the stale write fails closed without a lost update.
- **Invalid profile configuration** (duplicate gate name, unknown context reference, unparseable predicate, unsupported schema version): fails validation closed (exit `1`) before any gate runs. (An ordering-cycle is not expressible in the v1 linear ordered-list schema; that diagnostic is reserved for a future ordering-dependency hint.)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: SpecOps MUST provide a **versioned, stack-neutral gate-profile configuration** under the SpecOps namespace in `.specify` (consistent with the Feature 008 context-map location convention) that declares an **ordered** set of gates; each gate carries a stable **name**, the **command** it runs, an **applicability** predicate, a **timeout** (a positive integer number of **seconds**), and a **required-status** classification (`required` | `optional`) that **determines its failure semantics** — a `required` gate's non-zero exit blocks the verdict, an `optional` gate's is recorded but never blocks (the two are a single setting, not independently contradictory attributes). The **required-status** classification defaults to `required` when a gate omits it. The **timeout** defaults to a documented, stack-neutral fixed value when an **authored** gate omits it; the default MUST be a constant, never derived from wall-clock or environment. The **synthesized default profile** (FR-005), by contrast, MUST use **no** timeout (unbounded), so an upgraded repository whose `lint`/`test` command legitimately runs long is **not** newly terminated — preserving the pre-feature behavior (FR-005). The configuration MUST embed an explicit schema version (`output_version`).
- **FR-002**: Each gate MUST declare a **single applicability predicate** combining any of: `always` (global, unconditional), one or more **context ids** (Feature 008), one or more **changed-path** glob patterns, and a **risk** condition. The predicate MAY reference context ids from the map, and a resolved context's own `gates` list (Feature 008) MUST be honored as an **implicit context-id match** for the gates it names — so authors use one selection language whether or not a map exists. A **risk** condition MUST match by **named-key presence/equality**: it is satisfied when a resolved context's free-form `risk` mapping contains the named key (optionally equal to a declared value); SpecOps MUST NOT impose an ordinal risk scale or otherwise interpret the mapping (Constitution Principle V). Applicability MUST be evaluated deterministically against the resolved contexts, their risk mappings, and the changed paths (Feature 009) so that identical inputs always select the identical gate set and order.
- **FR-003**: SpecOps MUST **select** the applicable gates for a run deterministically and MUST record, for **every** declared gate, a specific machine-readable **reason** — drawn from the defined closed set `always`, `matched-context`, `matched-gate-ref`, `matched-path`, `matched-risk-key`, `out-of-scope` — for why it applied or was skipped. The selection MUST be reproducible byte-for-byte for identical configuration and inputs.
- **FR-004**: The selected required gates MUST run in their **declared relative order** (the ordering applies to the selected subset — skipped gates do not perturb it); the first required-gate failure MUST fail the verdict closed (Constitution Principle VI). A gate's non-zero exit MUST be interpreted per its declared **failure semantics** — a `required` failure blocks, an `optional` failure is recorded but does not block.
- **FR-005**: When a repository declares **no** gate profiles — an **absent** config file **or** a present config whose `profiles` list is **empty** — SpecOps MUST synthesize an **implicit default profile** from the legacy `test_command` and optional `lint_command` in `specops.json`, so verification runs unchanged on an upgraded repository (roadmap Rule 5); it MUST NOT silently run zero gates. The synthesized default profile MUST preserve the gate names `lint`/`test` **and their pre-feature timing behavior (no timeout — unbounded)**, so a consumer of the existing gate names, and a repository whose suite runs long, sees no regression until a custom profile is deliberately authored. The legacy commands MUST remain a supported configuration source. A `gate-profiles.yaml` that is **present but invalid** MUST NOT silently fall back to the default suite — `specops review` MUST fail closed (never a false pass that skips declared required gates); the read-only `gate list`/`validate` surfaces remain lenient.
- **FR-006**: SpecOps MUST record every gate result and every task/finding evidence link as a **structured evidence object** carrying: a **stable cache-key-derived id** (FR-009), the **producer** (gate/command identity plus the SpecOps CLI version), the exact **command**, its **exit code**, a **timezone-aware timestamp**, the **commit range** covered (`baseline..HEAD`, or a single commit sha when no baseline exists), the **affected paths**, a concise **summary**, and an **optional deterministic content digest** of a captured local artifact. The object MUST be a versioned, additive extension of the Feature 006 ledger and MUST be written atomically and interruption-safely through the SpecOps CLI (never hand-edited), guarded by Feature 006's concurrency control.
- **FR-007**: SpecOps MUST provide a **forward migration** from the legacy `<CLASS>:<summary>[; …]` evidence strings (classes `CLI_LOG | TEST_REPORT | SCREENSHOT_PATH | CODE_DIFF`) to structured evidence records, incrementing the Feature 006 ledger schema version, preserving each string's class and summary without loss, **retaining** the legacy string field alongside the new structured record (never replacing it, so Feature 010/011 rendering is unaffected), and keeping prior ledger shapes readable. The migration MUST be **idempotent** — re-running it yields byte-identical state. A migration test MUST cover every supported prior evidence shape. A legacy evidence string that does **not** match the `<CLASS>:<summary>` grammar MUST be preserved verbatim as an opaque migrated record (never dropped or rejected); a migration that cannot complete MUST leave the prior valid ledger readable (atomic, interruption-safe via Feature 006), never a partially-migrated state.
- **FR-008**: SpecOps MUST classify every gate in a verdict with exactly one value of the fixed **outcome taxonomy**: `required`, `optional`, `skipped`, `cached`, `failed`, or `unavailable`. `unavailable` (missing command or absent tool) MUST be distinct from `failed` (ran and did not satisfy its required status, including timeout).
- **FR-009**: SpecOps MUST reuse a prior evidence record as **`cached`** only when its **command**, its **inputs** (affected paths / commit range), the **context-map digest** (Feature 008), and the **commit** all still match; if any one differs, the cache MUST NOT be reused and the gate MUST run fresh. The evidence record's **id MUST be derived deterministically from this cache key** (producer / command / commit range / affected paths / context-map digest), so identical production yields an identical id and a changed cache key yields a **new** record under a **new** id that **supersedes** (never mutates in place) the prior immutable evidence record. The cache reuse MUST be evaluated over evidence **already recorded** in the ledger by a state-changing path (`complete-task`, `handoff finding fix`); to preserve `specops review`'s byte-for-byte **read-only** contract (Feature 004), review MUST NOT itself persist gate-run evidence — it reports a gate `cached` when a matching non-superseded record exists and otherwise runs it fresh, surfacing the fresh record's id in the verdict. **Persisting gate-run evidence from review** — the path that would let a gate's own prior run cache-hit or produce a superseding record end-to-end — is **deferred to a later feature**; the cache-key/id/supersession mechanism ships here and is exercised at the evidence layer.
- **FR-010**: A gate whose declared **timeout** is exceeded MUST be terminated and reported `failed` with a timeout reason; for a `required` gate this fails the verdict closed. Timeouts MUST be enforced deterministically and MUST NOT depend on wall-clock nondeterminism in recorded output.
- **FR-011**: The selected profile suite MUST execute **within `specops review`**, replacing the fixed `lint`/`test` gates in its pipeline (`reconcile → [selected profile suite] → working-tree → drift`); SpecOps MUST NOT introduce a separate standalone gate runner (the read-only `specops gate list`/`validate`/`report` surfaces are inspection-only). SpecOps MUST make **every review verdict fully provenanced**: for each gate that ran or was skipped, the verdict MUST identify the gate, its outcome-taxonomy value, its applied/skipped reason, the **inputs it covered** (commit range and paths), and the **immutable evidence record** (referenced by its FR-009 id) that supports it (where one exists). This composes the existing review gate report; it does not replace Feature 006/010's review-cycle model.
- **FR-012**: SpecOps MUST provide a **stable JSON report** over the verification result — profiles, per-gate selection reasons, outcome-taxonomy values, covered inputs, and supporting evidence — every JSON output embedding an explicit **`output_version`** field (the JSON contract version; distinct from the ledger `schema_version` and the config `output_version`) so downstream consumers (Features 013–014) can bind to a versioned contract.
- **FR-013**: SpecOps MUST provide an **optional SARIF output adapter** that projects review findings (Feature 011) into schema-valid **SARIF 2.1.0** (the OASIS standard version) preserving each finding's rule, location, and severity mapping (`blocking → error`, `advisory → warning`). SARIF output MUST be **opt-in** (absent by default) and its absence MUST never be reported as a defect.
- **FR-014**: SpecOps MUST provide **read-only** gate-profile validation that fails closed (exit `1`) on a duplicate gate name, an unknown context reference, an unparseable applicability predicate, or an unsupported configuration schema version — each as a distinct, specific diagnostic — before any gate runs. (An **ordering-cycle** diagnostic is **reserved** for a future ordering-dependency hint; the v1 linear ordered-list schema cannot express a cycle, so it is not a v1 validation defect.)
- **FR-015**: All report, validation, and evidence-read commands MUST be **read-only** (no repository or ledger mutation, verifiable by before/after comparison). Only gate execution recording, evidence writes, and migration change state, and those MUST be atomic and interruption-safe (Feature 006).
- **FR-016**: Every gate/evidence command MUST expose the fixed exit-code taxonomy already established for SpecOps commands — `0` success (including the supported no-profile / legacy / degraded states), `1` blocking/fail-closed (a required-gate failure or `unavailable`; any validation defect), `2` usage/input error (invalid arguments or configuration input) — with the fine-grained outcome carried in a stable `status` field (Constitution Principle VI).
- **FR-017**: For identical ledger, configuration, and branch inputs, every gate/evidence/report command MUST produce **byte-for-byte identical** output (deterministic selection, ordering, evidence serialization, and report rendering), consistent with the determinism guarantees of Features 008–011. This guarantee is scoped to output rendered from **recorded state** (the ledger, configuration, and branch), **not** to re-execution of a client gate command: a client command's own nondeterministic output lives inside an evidence record's `summary`, while the record's identity (its cache-key-derived id), the selection, the outcome taxonomy, and all ordering remain byte-for-byte reproducible.
- **FR-018**: SpecOps MUST **not** interpret test-framework-specific result formats in the core: an exit code plus a captured summary is the contract. Any framework-specific parsing MUST be out of scope, and no gate result may depend on a particular test framework, language, or tool (Constitution Principle V).
- **FR-019**: SpecOps MUST **not** store remote artifacts: an artifact digest MUST reference **local** content only, recording the digest current at production time so a later change is detectable without any remote copy.
- **FR-020**: A SpecOps gate **profile** MUST remain a **deterministic verification-command suite** invoked as native Spec Kit `shell`/`command` steps that produces evidence — it MUST NOT overlap with, duplicate, or replace Spec Kit's human `gate` step or its capability system (roadmap Rule 8). SpecOps builds no engine, loop, resume, or human-gate primitive.
- **FR-021**: Output ordering — for the selected-gate list, the JSON report, and any evidence listing — MUST follow a **defined canonical sort key** (gate declared order, then gate name; evidence by producer then timestamp then commit range) so output is reproducible independent of insertion order; the canonical ordering rule is part of the output contract.
- **FR-022**: The English and Portuguese documentation for every new command and behavior MUST remain **behaviorally equivalent** (Global Definition of Done), so a reader of either language obtains the same profile schema, selection rules, outcome taxonomy, evidence fields, caching semantics, and exit-code contract.

### Key Entities *(include if feature involves data)*

- **Gate Profile**: A versioned, ledger-external configuration record declaring an ordered set of gates; the stack-neutral successor to the single global `test_command`/`lint_command` pair.
- **Gate**: One entry in a profile — a stable name, a command, an applicability predicate, a timeout, a required-status classification, and failure semantics.
- **Applicability Predicate**: The single deterministic condition per gate — any of `always`, context ids, changed-path globs, and a named-key `risk` match — that decides whether a gate applies to a run. It may reference map contexts by id, and a context's own `gates` list counts as an implicit context-id match.
- **Selection Reason**: The recorded, machine-readable justification for why each gate applied or was skipped for a given run.
- **Gate Outcome**: One of the fixed taxonomy `required | optional | skipped | cached | failed | unavailable` describing a gate's disposition in a verdict.
- **Structured Evidence Record**: A versioned, immutable ledger object carrying a stable cache-key-derived id, producer, command, exit code, timestamp, commit range, affected paths, summary, and optional artifact digest; the machine-checkable successor to the `<CLASS>:<summary>` string.
- **Cache Key**: The tuple (command, inputs, context-map digest, commit) that must match in full for a prior evidence record to be reused as `cached`; the evidence record's stable id is derived deterministically from this tuple.
- **Artifact Digest**: An optional digest of captured local output content, recorded at production time to make later change detectable without remote storage.
- **Verification Verdict**: The provenanced result of a run — the selected suite, each gate's outcome and reason, the inputs it covered, and its supporting evidence record.
- **SARIF Projection**: The opt-in standard-format export of review findings for external tooling.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For 100% of fixtures with identical configuration, contexts, risk, and changed paths, the selected gate set, its order, and the per-gate applied/skipped reason are byte-for-byte identical across runs.
- **SC-002**: 100% of gates in a verdict carry exactly one outcome-taxonomy value (`required | optional | skipped | cached | failed | unavailable`), and `unavailable` is never conflated with `failed` in any fixture.
- **SC-003**: Given evidence recorded in the ledger, a matching non-superseded record is reused as `cached` in 100% of fixtures where command, inputs, context-map digest, and commit all match, and reused in 0% of fixtures where any one differs (the gate runs fresh). The supersession mechanism — a changed cache key yields a new record under a new id that supersedes the prior — is exercised at the evidence layer; because `specops review` does not itself persist gate-run evidence (read-only), end-to-end gate self-caching depends on evidence recorded by a state-changing path (deferred per FR-009).
- **SC-004**: Every review verdict identifies, for 100% of gates that ran or were skipped, the gate, its outcome, its reason, the commit range and paths it covered, and its supporting immutable evidence record where one exists.
- **SC-005**: 100% of legacy `<CLASS>:<summary>` evidence strings migrate into structured evidence records with zero loss of class or summary, the ledger schema version is incremented, and 100% of pre-feature ledgers remain readable after migration.
- **SC-006**: A repository with **no** gate-profile configuration runs verification via the synthesized default profile in 100% of fixtures (no retroactive block, roadmap Rule 5), and a repository with an invalid profile configuration fails validation closed (exit `1`) with a distinct diagnostic in 100% of seeded-defect fixtures.
- **SC-007**: A `required` gate failure (non-zero exit, timeout, or `unavailable`) fails the verdict closed in 100% of fixtures, while an `optional` gate failure blocks in 0% of fixtures.
- **SC-008**: Every gate/evidence/report command returns within the fixed `0`/`1`/`2` exit-code taxonomy with a populated `status` field, never mutates repository or ledger state on a read-only path (verified by before/after comparison), and produces byte-for-byte identical output for identical inputs in 100% of fixtures.
- **SC-009**: The JSON report embeds an explicit `output_version` field and renders profiles, selection reasons, outcome-taxonomy values, covered inputs, and evidence in 100% of fixtures; enabling the SARIF adapter emits schema-valid SARIF preserving rule/location/severity in 100% of findings fixtures, and SARIF is emitted in 0% of runs that do not request it.
- **SC-010**: No gate result depends on a test framework, language, or remote store: 100% of gate results derive solely from a command, its exit code, a captured summary, and local-content digests, with framework-specific parsing absent from the core.

## Assumptions

- The versioned, concurrency-safe ledger from Feature 006 (currently schema v5 after Feature 011) is the storage for structured evidence; the new evidence object and any run/verdict metadata are additive, forward-migrated extensions written exclusively through the SpecOps CLI (Constitution Principle II), incrementing the schema version with a migration test and preserving read compatibility with prior shapes.
- Gate-profile **configuration** lives under the SpecOps namespace in `.specify` (following Feature 008's context-map placement) rather than inside `specops.json`, because a profile is richer than the flat `test_command`/`lint_command` keys; those keys remain a supported legacy source that synthesizes the implicit default profile (roadmap Rule 5).
- Profile applicability is a **single self-declared predicate per gate** (`always` / context ids / changed-path globs / named-key risk) that composes Feature 008's contexts and risk mappings and Feature 009's changed-path/impact resolution; a resolved context's own `gates` list is honored as an implicit context-id match. When no context map is present, context- and risk-keyed predicates match nothing while `always` and changed-path predicates still select, so a map-less repository still runs its verification and a missing map stays a supported reported state, never a defect.
- The selected profile suite executes **inside `specops review`**, replacing its fixed `lint`/`test` gates (`reconcile → suite → working-tree → drift`); this feature introduces **no** standalone gate runner, and the `specops gate` surfaces (`list`/`validate`/`report`) are read-only inspection only. Composing the existing review verdict (rather than a parallel surface) is what makes "every review verdict identifies which gates ran" achievable and sets up the Feature 017 `preflight` rename.
- Evidence continues to link at the same points it does today (task completion, and Feature 011 finding `FIXED`/`VERIFIED`); this feature restructures the evidence *representation* and migrates the string form, but does not change the finding lifecycle or the review-cycle model (that is Feature 011 / Feature 006).
- The core records an exit code plus a captured summary and, optionally, a digest of local captured output; it does not parse framework-specific result files and stores no remote artifacts — consistent with Domain Agnosticism (Principle V) and the feature's explicit non-goals.
- SARIF is an **output** adapter for review findings and is opt-in; the SARIF *input* adapter for external findings is Feature 015 and is out of scope here.
- SpecOps gate profiles are distinct from Spec Kit's native human `gate` step and capability system (roadmap Rule 8): profiles are verification-command suites invoked as native `shell`/`command` steps that produce evidence, and this feature builds no engine, loop, resume, or human-gate primitive.
- English and Portuguese documentation remain behaviorally equivalent, and every new surface degrades safely to a supported state when no profile configuration or context map exists (roadmap Rule 5).
- **`specops review` stays byte-for-byte read-only** (Feature 004's FR-007 contract): it evaluates the profile suite and reads existing evidence for cache decisions and provenance, but does **not** persist gate-run evidence. The cache-key/id/supersession machinery (FR-009) ships and is verified at the evidence layer; making a gate's *own* prior run cache-hit or supersede end-to-end requires review to persist evidence, which would narrow the read-only contract and is therefore **deferred to a later feature** (a natural fit for the Feature 013/014 diagnostics/lane work). This resolves the internal tension between FR-015 ("gate execution recording … change state") and the read-only review contract in favor of the latter for this feature.
