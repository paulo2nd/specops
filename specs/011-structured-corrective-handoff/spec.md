# Feature Specification: Structured Corrective Handoff

**Feature Branch**: `011-structured-corrective-handoff`

**Created**: 2026-07-22

**Status**: Draft

**Input**: User description: "Introduce structured corrective handoffs with stable finding IDs, severity, authorized paths, expected evidence, closure criteria, lifecycle states, and trace links. Keep Markdown revision reports as a rendered human interface and prevent approval while blocking findings remain unverified."

## Complement Boundary *(mandatory context)*

SpecOps runs **inside** Spec Kit and is a **complement**, never a replacement (Constitution Principle I). Today a review that rejects records its findings as free-form lines in `revisions/revision-X.md` — `[File]:[Line] - [rule violated and short action]` — and the corrective round that follows is described in ordinary prose. That prose is a human artifact, not machine-checkable state: a fresh session cannot deterministically answer "which findings are still open, which are fixed, which are verified, what paths was the correction authorized to touch, and what evidence closes each finding?" from repository state alone. Feature 006 already gives a versioned, concurrency-safe ledger with `review_cycles` (each round carrying `round`, `started_at`, `completed_at`, `result`, and context provenance), and Feature 010 already links findings to a **path** (via the `[File]` token) and to their **review cycle**, and links corrections via the **corrective commits** recorded in that cycle — but explicitly **without** any per-finding identifier, deferring that anchoring to this feature.

This feature promotes review findings and correction authorization to first-class, versioned **ledger state**, contributing only:

1. A **structured finding** record with a stable ID, severity, the rule violated, a location, and a concise action — the machine-checkable successor to the `revision-X.md` prose line.
2. A **corrective handoff** record — bound to the originating Feature 006 review cycle — that authorizes the shared corrective paths for its findings, each of which carries its own expected evidence and closure criteria.
3. A **finding lifecycle** — `OPEN → FIXED → VERIFIED` — with monotonic, fail-closed transitions that link each correction to its task(s), commit(s), evidence, and originating cycle.
4. A **blocking-approval invariant**: a review cannot be recorded `APPROVED` and the feature cannot enter `DONE` while any **blocking** finding is unverified — composing Feature 006's existing review-cycle/`DONE` gate rather than replacing it.
5. **Handoff validation and reports** (human-readable and stable JSON) that detect dangling links, unresolved closure, and contradictory finding state, failing closed with specific diagnostics.
6. A **rendered Markdown revision report**: `revisions/revision-X.md` remains the human-facing interface, now *rendered deterministically from* the structured state (which is authoritative), preserving compatibility for existing revision-report consumers.

Consistent with roadmap Rule 8 and the explicit non-goals below, this feature builds **no** Spec Kit workflow engine, gate, loop, or resume mechanism; findings never modify product code; it introduces **no** issue-tracker integration and **no** parallel correction ownership. It does **not** redesign the evidence/test-result artifact (that is Feature 012 — here evidence is linked as the existing `<CLASS>:<summary>` record), and it does **not** add new gate profiles. It composes Feature 006's ledger, concurrency control, and review-cycle model and Feature 010's finding↔path↔cycle linkage and drift/authorized-path concept, adding only the structured finding, the corrective handoff, the lifecycle, the blocking-approval invariant, validation, reports, and the Markdown rendering.

## Clarifications

### Session 2026-07-22

- Q: At what level are closure criteria and expected evidence owned — per-finding, per-handoff, or both? → A: **Per-finding**. Each finding carries its **own** closure criteria and expected evidence, so every blocking finding is independently verifiable — exactly what the `OPEN → FIXED → VERIFIED` lifecycle and the "no approval while any blocking finding is unverified" invariant require. The handoff owns only the shared **authorized corrective paths** and the originating-cycle binding; validation and the `VERIFIED` gate evaluate each finding's closure independently.
- Q: How does a blocking finding become `VERIFIED` — mechanical CLI decision, auto-verify, or reviewer assertion? → A: **Mechanical precondition + reviewer assertion**. The CLI fail-closed guards that the finding's expected evidence is present and its links (task/commit/evidence) resolve before permitting `VERIFIED`; the reviewer's act of recording the transition during the review pass *is* the closure judgment. The CLI enforces the deterministic guard (Principle II) but never semantically judges whether the evidence "really" closes the finding (Principle IV owns that judgment). There is **no** auto-verify: evidence presence alone never transitions a finding.
- Q: What severity model does a finding use — two-value, richer + configurable threshold, or richer + fixed mapping? → A: **Two-value** (`blocking` | `advisory`). A finding's severity is exactly one of `blocking` or `advisory`; only `blocking` findings gate approval (FR-007), and `advisory` findings are retained and reported but never block. This is the minimal model the "prevent approval while blocking findings remain unverified" invariant requires; a finer taxonomy can be introduced later as an additive schema change if a success criterion ever needs it.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Record a resumable corrective handoff at rejection (Priority: P1)

A reviewer rejects a review. Instead of writing loose prose, SpecOps records a **corrective handoff** bound to the current review cycle: for each non-conformity a **structured finding** carrying a stable ID, a severity (`blocking` or `advisory`), the rule violated, a location (path and optional line), a concise action, and — per finding — its own **expected evidence** and **closure criteria**; and, for the handoff as a whole, the shared **authorized corrective paths** the fix is expected to touch. All of it is written atomically to the Feature 006 ledger through the CLI. A fresh session — with no memory of the review — can then reconstruct the exact rejected-review state (which findings are open, at what severity, what may be changed, and what evidence is required) from repository state alone.

**Why this priority**: This is the resumability half of the acceptance gate — "a rejected review can be resumed by a fresh session using only repository state." It is the foundation every other slice builds on, and it is independently valuable even before the hard approval gate exists, because it turns an unstructured revision report into deterministic, machine-readable state.

**Independent Test**: Reject a review, record a handoff with several findings of mixed severity — each with its own expected evidence and closure criteria — plus the handoff's shared authorized corrective paths; discard all session context; then, from repository state alone, read back every finding with its stable ID, severity, rule, location, action, expected evidence, and closure criteria, plus the handoff's authorized paths — confirming each field round-trips, the write was atomic, the handoff is bound to the correct review cycle, and repeated reads are byte-for-byte identical.

**Acceptance Scenarios**:

1. **Given** a review at `REJECTED` with several non-conformities, **When** the reviewer records the corrective handoff, **Then** each finding is persisted with a stable ID unique within the feature, a severity (`blocking` or `advisory`), a rule, a location, and a concise action, and the handoff is written atomically and bound to the originating review cycle round.
2. **Given** a recorded handoff, **When** a fresh session with no prior context reads repository state, **Then** it recovers every finding's ID, severity, rule, location, state, expected evidence, and closure criteria, plus the handoff's shared authorized corrective paths, with zero reliance on prior-session memory.
3. **Given** the same rejected review, **When** the handoff is recorded and then re-read repeatedly, **Then** finding IDs, ordering, and serialization are byte-for-byte identical across reads.
4. **Given** an interrupted handoff write, **When** the ledger is next read, **Then** the previous valid ledger remains readable and no partial handoff is observed (atomic, interruption-safe via Feature 006).
5. **Given** a review with **zero** non-conformities, **When** the outcome is recorded, **Then** no corrective handoff is created and approval is not blocked by this feature.

### User Story 2 - Drive findings to verified and gate approval (Priority: P2)

An implementer resolves the findings. For each one they mark it `FIXED`, linking the task, the commit(s), and the evidence that satisfies its closure criteria; a subsequent review pass marks satisfied findings `VERIFIED`. The state machine is monotonic and fail-closed: a finding cannot skip from `OPEN` straight to `VERIFIED`, cannot move backward, and cannot be `VERIFIED` without the expected evidence linked. Approval is impossible while any **blocking** finding is unverified — the review cannot be recorded `APPROVED` and the feature cannot enter `DONE` — while **advisory** findings never block. This composes, and never bypasses, Feature 006's existing review-cycle/`DONE` gate.

**Why this priority**: This is the enforcement half of the acceptance gate — "approval is impossible until all blocking findings have verified evidence." It is a distinct, independently testable slice that operates over seeded finding records and delivers the safety guarantee, but it depends on the state model from US1, so it is P2.

**Independent Test**: From fixtures seeding findings in each state, attempt every transition: confirm `OPEN → FIXED` requires a task/commit/evidence link, `FIXED → VERIFIED` requires the expected evidence and cannot be reached from `OPEN`, backward transitions fail closed, and approval/`DONE` is blocked while any blocking finding is unverified and permitted once all blocking findings are `VERIFIED`; confirm advisory-only open findings never block.

**Acceptance Scenarios**:

1. **Given** an `OPEN` blocking finding, **When** the implementer marks it `FIXED` with a task, commit, and evidence, **Then** the transition is recorded atomically and the finding links to that correction and its originating cycle.
2. **Given** a `FIXED` finding whose expected evidence is present, **When** a review pass marks it `VERIFIED`, **Then** the transition succeeds and the finding is closed.
3. **Given** an `OPEN` finding, **When** a `VERIFIED` transition is attempted directly (skipping `FIXED`), **Then** the command fails closed (exit `2`) and the finding state is unchanged.
4. **Given** any finding, **When** a backward transition (e.g. `VERIFIED → OPEN`) or a `VERIFIED` with no linked evidence is attempted, **Then** the command fails closed (exit `2`) and records nothing.
5. **Given** a review cycle with at least one unverified **blocking** finding, **When** approval or a `DONE` transition is attempted, **Then** it is blocked and the specific unverified blocking findings are named.
6. **Given** a review cycle whose every blocking finding is `VERIFIED` (advisory findings may remain open), **When** approval is recorded, **Then** it succeeds and `DONE` becomes reachable.
7. **Given** a handoff whose blocking findings are all `VERIFIED`, **When** the handoff is closed, **Then** closure is recorded (exit `0`), is idempotent on re-close, and is the precondition that unblocks approval; a close attempted while any blocking finding is unverified fails closed (exit `1`) and names them.
8. **Given** blocking findings spread across an earlier and a later review round, **When** approval is attempted with the earlier round's finding still unverified, **Then** it is blocked — the invariant is evaluated over all rounds (feature-global), not only the latest.

### User Story 3 - Validate and report the corrective handoff (Priority: P2)

A reviewer or auditor asks SpecOps for the state of the corrective work: every handoff, its findings, their states and severities, the links from each correction to its task/commit/evidence, and the remaining blocking findings. SpecOps emits a concise human-readable report and a stable JSON contract. A companion validation pass fails closed on a dangling link (a finding referencing a missing cycle, task, commit, or evidence), a blocking finding missing its closure criteria, a contradictory state (a `VERIFIED` finding with no linked evidence, or a `FIXED` finding with no commit/task), or a duplicate finding ID — producing a distinct diagnostic per defect.

**Why this priority**: The report and validation are the auditability payload — they make the handoff state inspectable and prove its integrity — and they satisfy the "resumed using only repository state" property by surfacing exactly what remains. They build on the same ledger state as US1/US2 and are independently testable against fixtures, so they are P2.

**Independent Test**: Against a fixture with a well-formed handoff, run the report and confirm every finding resolves to its full chain and validation exits `0`; against fixtures seeded with a dangling link, a missing closure criterion, a contradictory state, and a duplicate ID, confirm validation exits `1` with one specific diagnostic per defect and that the JSON output is stable and byte-for-byte reproducible.

**Acceptance Scenarios**:

1. **Given** a well-formed handoff whose every finding has valid links and (for blocking findings) closure criteria, **When** validation runs, **Then** it reports success (exit `0`) and the report renders each finding's full chain (severity → rule → location → state → task/commit/evidence).
2. **Given** a finding referencing a task, commit, or evidence absent from the ledger or branch, **When** validation runs, **Then** it fails closed (exit `1`) and reports the dangling reference (deferring commit-existence enforcement to `specops reconcile` while surfacing the trace-level break).
3. **Given** a blocking finding with no closure criteria, **When** validation runs, **Then** it fails closed (exit `1`) and names that finding as missing closure criteria.
4. **Given** a `VERIFIED` finding with no linked evidence, or a `FIXED` finding with no commit/task, **When** validation runs, **Then** it fails closed (exit `1`) and reports the contradictory state.
5. **Given** identical ledger and branch inputs, **When** the JSON handoff report is produced repeatedly, **Then** its content and ordering are byte-for-byte identical across runs.

### User Story 4 - Render compatible revision reports and migrate legacy prose (Priority: P3)

Existing consumers still expect a human-readable `revisions/revision-X.md`. SpecOps renders that file deterministically **from** the structured handoff state, so the ledger is authoritative and the Markdown is a projection rather than the source of truth. Repositories that already hold legacy `revision-X.md` prose (short-format `[File]:[Line] - action` lines with no structured state) are treated as a supported prior shape: those files remain readable, are never reported as defects, and can be imported into structured findings without loss. Feature 010's trace re-sources findings from the structured state when it is present, using the new stable finding IDs, and falls back to legacy parsing otherwise, with no regression to the Feature 010 report contract.

**Why this priority**: Backward compatibility and the Markdown human interface matter for adoption and audit continuity but are not required to prove the core acceptance gate, so this is P3. It is independently testable by rendering from a fixture handoff and by reading a legacy revision fixture.

**Independent Test**: From a structured handoff fixture, render `revision-X.md` and confirm it is byte-for-byte deterministic, human-readable, and compatible with the prior format; from a legacy revision-prose fixture with no structured state, confirm it reads without error, is never flagged as a defect, and imports into structured findings with every legacy line preserved; run the Feature 010 trace against both and confirm findings resolve to stable IDs when structured and via legacy parsing otherwise, with the 010 fixtures still passing.

**Acceptance Scenarios**:

1. **Given** a structured handoff, **When** the revision report is rendered, **Then** `revisions/revision-X.md` is produced deterministically from the ledger state, remains human-readable, and preserves compatibility with the prior revision-report format.
2. **Given** a repository with only legacy revision prose and no structured findings, **When** any handoff read command runs, **Then** the legacy files read without error and are never reported as defects (roadmap Rule 5).
3. **Given** a legacy revision file, **When** it is imported, **Then** every legacy finding line is preserved as a structured finding with no loss of location or action text.
4. **Given** a ledger with structured findings, **When** the Feature 010 trace report runs, **Then** it resolves findings to their stable IDs from the structured state; **Given** a legacy ledger, **When** the same report runs, **Then** it falls back to parsing `revision-X.md`, and all Feature 010 fixtures continue to pass.

### Edge Cases

- **Review with zero findings**: no corrective handoff is created; the blocking-approval invariant imposes no additional constraint and Feature 006's existing cycle gate governs approval unchanged.
- **Advisory-only open findings**: approval and `DONE` are permitted; advisory findings are retained and reported but never block (0% false block).
- **Legacy ledger with no structured findings**: the blocking-approval invariant has no structured finding state to enforce and **degrades** to Feature 006's existing review-cycle/`DONE` gate — an upgraded repository whose rejection predates this feature is never retroactively blocked (roadmap Rule 5); explicit `validate`/transition commands remain strict.
- **Re-entering review after corrections**: findings carry forward across rounds — `VERIFIED` findings stay verified, `FIXED`-but-unverified findings are re-examined, and newly discovered non-conformities are added as new findings with new IDs in the new round.
- **Illegal transition** (`OPEN → VERIFIED` skip, any backward move, `VERIFIED` without evidence, `FIXED` without commit/task): fails closed (exit `2`) leaving state unchanged.
- **Duplicate finding ID**: rejected at creation and reported by validation as a distinct defect; IDs are unique within the feature. A recurrence of the same `file:line` in a later round is a **new** finding with a **new** ID, not a duplicate.
- **Finding whose location path was later removed**: still valid; the closure criteria may legitimately reference the removal, and the finding is never invalidated solely because its path left the tree.
- **Correction touches a path outside the handoff's authorized corrective paths**: surfaced through Feature 010's drift classification (the path is `unexplained` unless planned or acknowledged) rather than by a new gate here; this feature records the authorization, it does not re-implement the drift gate.
- **`VERIFIED` finding whose fix commit is absent from the active branch**: reported as a dangling reference by validation, deferring authoritative commit-existence enforcement to `specops reconcile` (mirroring Feature 010).
- **Concurrent transition of the same finding by two sessions**: guarded by Feature 006's existing concurrency control (lock / compare-and-swap); the stale write fails closed without a lost update.
- **Pre-feature ledger read**: task/review records lacking finding/handoff fields are a valid supported prior shape via additive forward migration; reads never fail and absent fields are reported explicitly, not as defects.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: SpecOps MUST record a **structured finding** carrying a **stable ID** (unique within the feature and unchanged across re-renders, resumes, and fresh sessions), a **severity** (`blocking` or `advisory` — see FR-003), the **rule** violated, a **location** (a repository-relative path with an optional line), a **concise action**, and — owned **per finding** — its own **closure criteria** and an **expected-evidence descriptor**. The expected-evidence descriptor is the **declared expectation**, recorded at finding creation, of what will close the finding (a class/kind hint or short expectation string); it is distinct from the **actual** `<CLASS>:<summary>` evidence record linked later at `FIXED` (FR-005), which is what the `VERIFIED` precondition checks (FR-006). Finding IDs MUST be assigned deterministically so that identical inputs produce identical IDs and ordering.
- **FR-002**: SpecOps MUST record a **corrective handoff** bound to the **originating review cycle round** (Feature 006 `review_cycles`), carrying the shared **authorized corrective paths** for the round; each handoff contains the round's structured findings, and closure criteria and expected evidence are owned per finding (FR-001), not at the handoff level. The handoff and its findings MUST be written atomically and interruption-safely through the Feature 006 CLI surface (never hand-edited or held in agent memory), guarded by Feature 006's existing concurrency control; a concurrent or stale write MUST fail closed without a lost update.
- **FR-003**: A finding's **severity** MUST be exactly one of the two labels `blocking` or `advisory` (a closed set — an unrecognized severity is a usage error, exit `2`). Only `blocking` findings gate approval (FR-007); `advisory` findings MUST be retained and reported and MUST NOT block approval.
- **FR-004**: A finding MUST move through the monotonic lifecycle **`OPEN → FIXED → VERIFIED`**. Skipping a state (e.g. `OPEN → VERIFIED`), any backward transition, and any transition that violates its precondition MUST fail closed (exit `2`) and leave the prior state unchanged.
- **FR-005**: The **`FIXED`** transition MUST link the finding to the resolving **task**, its **commit(s)**, and the **actual evidence** that satisfies its closure criteria (linked as the existing `<CLASS>:<summary>` record — no evidence redesign), and MUST link the correction to the originating review cycle. This actual linked evidence is what the `VERIFIED` precondition (FR-006) checks against the finding's declared expected-evidence descriptor (FR-001). A `FIXED` finding with no linked commit and task is a contradictory state (FR-010).
- **FR-006**: The **`VERIFIED`** transition MUST be recorded during a review pass and MUST NOT be reachable from `OPEN` (a finding cannot be verified before it is fixed). The CLI MUST fail-closed guard the transition on a **mechanical precondition** — the finding's **actual linked evidence** (FR-005) is present and its task/commit/evidence links resolve, satisfying the presence declared by the finding's expected-evidence descriptor (FR-001) — and MUST NOT auto-verify: satisfying the precondition never transitions a finding by itself. A transition attempted when the precondition is unmet MUST fail closed with a **usage error (exit `2`)** and leave the finding state unchanged. The reviewer's act of recording `VERIFIED` during the review pass **is** the closure judgment (Principle IV); the CLI enforces the deterministic guard (Principle II) but MUST NOT semantically judge whether the evidence adequately closes the finding.
- **FR-007**: SpecOps MUST enforce a **blocking-approval invariant**: a review cycle MUST NOT be recorded `APPROVED`, and the feature MUST NOT enter `DONE`, while any `blocking` finding is not `VERIFIED` (equivalently, while the round's corrective handoff is not closable per FR-023). The attempt MUST fail closed and name the specific unverified blocking findings. This invariant MUST **compose** Feature 006's existing review-cycle/`DONE` gate, never bypass or duplicate it.
- **FR-008**: When a review cycle has no structured findings (a legacy ledger, or an approval with zero findings), the blocking-approval invariant MUST **degrade** to Feature 006's existing gate rather than block — an upgraded repository whose rejection predates this feature MUST NOT be retroactively blocked (roadmap Rule 5). Explicit `validate` and transition commands MUST remain strict.
- **FR-009**: The handoff MUST record the **authorized corrective paths** as the paths the correction is expected to touch, for audit and for Feature 010 drift classification. SpecOps MUST NOT re-implement the drift gate: a corrective change touching an unauthorized path surfaces through Feature 010's `unexplained` classification (unless planned or acknowledged), not through a new gate in this feature.
- **FR-010**: SpecOps MUST provide read-only **handoff validation** that fails closed (exit `1`) on: (a) a **dangling reference** (a finding referencing a review cycle, task, commit, or evidence that cannot be resolved), (b) a **blocking finding missing closure criteria**, (c) a **contradictory state** (a `VERIFIED` finding with no linked evidence, or a `FIXED` finding with no commit/task), and (d) a **duplicate finding ID**. Each defect MUST be reported as a distinct, specific diagnostic.
- **FR-011**: For commit references, validation MUST surface the trace-level break as a dangling reference but MUST defer authoritative commit-existence enforcement to `specops reconcile` (Constitution Principle II), rather than duplicating or contradicting that check.
- **FR-012**: SpecOps MUST provide a **handoff report** in both a concise human-readable form and a stable JSON contract, each rendering every handoff, its findings (ID, severity, rule, location, state), the link from each correction to its task/commit/evidence and originating cycle, and the set of remaining unverified blocking findings. Every JSON output MUST embed an explicit **schema-version** field so consumers (Features 012–014) can bind to a versioned contract.
- **FR-013**: SpecOps MUST author findings and the corrective handoff as **structured ledger state first**, during the review phase and through the CLI (never by hand-editing Markdown), and MUST render the human-readable **`revisions/revision-X.md`** deterministically **from** that authoritative structured state (Principle II, repo-as-state). The rendered Markdown is a **downstream projection**, never the source of truth — the sole exception is the legacy prose of FR-014, which has no structured origin. The rendered report MUST remain compatible with prior revision-report consumers and MUST be byte-for-byte reproducible from identical state.
- **FR-014**: SpecOps MUST treat existing **legacy revision prose** (`revision-X.md` short-format `[File]:[Line] - action` lines with no structured state) as a valid supported prior shape: it MUST read without error, MUST NOT be reported as a defect, and MUST be importable into structured findings preserving every legacy line's location and action text without loss.
- **FR-015**: Feature 010's trace MUST re-source findings from the structured state when present — resolving each finding to its **stable ID** (the per-finding identifier deferred in Feature 010) — and MUST fall back to parsing `revision-X.md` for legacy ledgers, without regressing the Feature 010 report contract (the change is additive).
- **FR-016**: All handoff read commands (report, validation, and any finding/handoff read) MUST be **read-only**: they MUST NOT mutate repository or ledger state, verifiable by before/after comparison. Only handoff creation, finding transitions, handoff closure, and round carry-forward (FR-002, FR-004–FR-007, FR-023, FR-024) change state.
- **FR-017**: Every handoff command MUST expose the fixed exit-code taxonomy already established for SpecOps commands — `0` success (including the supported no-findings and legacy states), `1` blocking/fail-closed (any validation defect; a blocked approval), `2` usage/input error (illegal transition, conflicting or invalid input, bad arguments) — with the fine-grained outcome carried in a stable `status` field.
- **FR-018**: For identical ledger and branch inputs, every handoff command MUST produce **byte-for-byte identical** output (deterministic ID assignment, ordering, and serialization), consistent with the determinism guarantees of Features 008–010.
- **FR-019**: The finding, handoff, and lifecycle state MUST be a **versioned, additive** extension of the Feature 006 ledger. Adding these fields MUST **increment the Feature 006 ledger schema version** and be covered by a forward-migration test that upgrades a pre-feature ledger (task/review records lacking finding/handoff fields) without data loss; reads of prior ledger shapes MUST NOT fail or be reported as defects. The new state MUST **extend, and never invalidate, the existing `review_cycles` records** (each handoff binds to a cycle round by reference): a v4 ledger's cycle records MUST remain readable and semantically unchanged after migration.
- **FR-020**: A rejected review MUST be fully **resumable from repository state alone**: every open blocking finding — its severity, its expected evidence, and its closure criteria — together with the handoff's shared authorized corrective paths MUST be reconstructable by a fresh session from the ledger without any reliance on prior-session memory.
- **FR-021**: SpecOps MUST NOT modify product code as part of a finding or correction (findings only describe and link), MUST NOT introduce issue-tracker integration, and MUST NOT support parallel/concurrent correction ownership — corrective transitions are serialized by Feature 006's concurrency control.
- **FR-022**: SpecOps MUST NOT introduce a language- or framework-specific parser; findings and links MUST derive only from repository-relative paths, rule and action text, the Feature 006 ledger, and Feature 010 links, keeping the core stack-neutral (Constitution Principle V).
- **FR-023**: SpecOps MUST provide an operation to **close a corrective handoff** that succeeds only when every `blocking` finding in that handoff is `VERIFIED` (advisory findings MAY remain open) and MUST fail closed (exit `1`) otherwise, naming the unverified blocking findings. A closed handoff is the precondition the blocking-approval invariant (FR-007) checks before a review cycle may be recorded `APPROVED`. Closing MUST be **recorded** (auditable, never a deletion) and MUST be **idempotent** — re-closing an already-closed handoff is a supported no-op (exit `0`).
- **FR-024**: When a new review round opens after a corrective round, SpecOps MUST **carry findings forward deterministically**: `VERIFIED` findings retain their state and stable ID; `FIXED`-but-unverified and `OPEN` findings remain open for re-examination under their existing IDs; and a newly observed non-conformity — even at a previously seen `file:line` — MUST be recorded as a **new** finding with a **new** ID (never a silent mutation of an existing one). The blocking-approval invariant (FR-007) MUST be evaluated over **all** carried-forward and new blocking findings across every round (feature-global), not only the latest round.
- **FR-025**: Every finding/handoff output — human report, JSON, validation, and rendered `revisions/revision-X.md` — MUST order findings by a **defined canonical sort key** (round, then severity, then location, then finding ID) so output is reproducible independent of insertion order; the canonical ordering rule is part of the output contract, not an implementation incidental.
- **FR-026**: The English and Portuguese documentation for every new command and behavior MUST remain **behaviorally equivalent** (Global Definition of Done), so a reader of either language obtains the same finding lifecycle, gate semantics, and exit-code contract.

### Key Entities *(include if feature involves data)*

- **Corrective Handoff**: A versioned, ledger-stored record bound to a Feature 006 review cycle round, authorizing the shared corrective paths for the findings raised in that round (closure criteria and expected evidence are owned per finding); the container the corrective work is resumed from and that is **closed** (FR-023) once its blocking findings are all `VERIFIED`.
- **Finding**: A first-class, versioned record of a single non-conformity, carrying a stable ID, severity, the rule violated, a location, a concise action, a per-finding closure criteria, an expected-evidence descriptor, and a lifecycle state; the machine-checkable successor to the `revision-X.md` prose line.
- **Severity**: The closed classification `blocking` | `advisory` that determines whether an unverified finding gates approval.
- **Finding State**: One of `OPEN`, `FIXED`, `VERIFIED` — the monotonic lifecycle whose `VERIFIED` terminus, for every blocking finding, is the precondition for approval and for handoff closure.
- **Closure Criteria**: The recorded condition, owned **per finding**, under which that finding may become `VERIFIED`; the per-finding target the correction must satisfy, expressed via the finding's expected-evidence descriptor.
- **Expected-Evidence Descriptor**: The per-finding declared expectation (recorded at creation) of what evidence will close the finding; distinct from the actual `<CLASS>:<summary>` evidence linked at `FIXED` that the `VERIFIED` precondition checks.
- **Correction Link**: The linkage from a `FIXED`/`VERIFIED` finding to its resolving task, commit(s), evidence, and originating review cycle.
- **Handoff Defect**: A specific, machine-checkable finding produced by validation — a dangling reference, missing closure criteria, a contradictory state, or a duplicate ID — each with enough locating detail to act on.
- **Rendered Revision Report**: The human-readable `revisions/revision-X.md` projection of the structured handoff state, retained for compatibility and audit but no longer the source of truth.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of findings receive a stable ID that is unique within the feature and unchanged across repeated renders, resumes, and fresh sessions; identical inputs produce byte-for-byte identical IDs and ordering.
- **SC-002**: A rejected review is 100% reconstructable by a fresh session from repository state alone — every open blocking finding, its severity, its expected evidence, and its closure criteria, plus the handoff's shared authorized corrective paths, are recovered with zero reliance on prior-session memory.

## Assumptions

- The versioned, concurrency-safe ledger from Feature 006 (currently schema v4) is the storage for findings, handoffs, and lifecycle state; these are additive, forward-migrated extensions written exclusively through the SpecOps CLI (Constitution Principle II), preserving read compatibility with prior ledger shapes and incrementing the schema version with a forward-migration test.
- Severity is limited to `blocking` and `advisory` because the roadmap's requirement is precisely "prevent approval while blocking findings remain unverified"; a finer severity taxonomy is not required and is out of scope for this feature.
- The finding lifecycle is `OPEN → FIXED → VERIFIED`; `VERIFIED` is asserted during a subsequent review pass and does not, by itself, require a different human actor than the one who marked `FIXED` — the roadmap's "no parallel correction ownership" is satisfied by serializing transitions through Feature 006's concurrency control, not by modeling actor identity.
- Evidence is linked as the existing machine-collected `<CLASS>:<summary>` record (Constitution Principle III); the structured-evidence and gate-profile redesign is Feature 012 and is out of scope here.
- Findings are linked to paths and review cycles using Feature 010's existing linkage, and the **authorized corrective paths** compose Feature 010's drift classification rather than introducing a new gate; commit-existence enforcement remains owned by `specops reconcile` (Constitution Principle II).
- The `revisions/revision-X.md` report remains the human-facing interface but becomes a deterministic **rendering** of the authoritative structured state; existing repositories that hold only legacy revision prose are a supported prior shape and are never retroactively blocked (roadmap Rule 5).
- Feature 010's trace is extended additively to consume the new stable finding IDs when present and to fall back to legacy parsing otherwise; its existing report contract and fixtures are preserved.
- English and Portuguese documentation remain behaviorally equivalent, and every new surface degrades safely to a supported state when no structured findings exist (roadmap Rule 5).
